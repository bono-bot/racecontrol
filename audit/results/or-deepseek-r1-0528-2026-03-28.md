# OpenRouter MMA: deepseek/deepseek-r1-0528

### Audit Findings

#### **P1 (Critical/Security/Data Loss) Issues**

1. **ID: P1-001**  
   **Severity**: P1  
   **Category**: Race Condition / Data Corruption  
   **Location**: `billing.rs` (`start_billing_session`)  
   **Description**:  
   - The in-memory `active_timers` check (read lock) and subsequent DB insert lack atomicity. The `UNIQUE` DB constraint prevents duplicates, but the error is not explicitly handled. If two concurrent requests pass the in-memory check, one will fail at the DB layer, but the error propagates as a generic string. This leaves the in-memory state (`active_timers` and `pods`) inconsistent if the DB fails after the in-memory update.  
   **Impact**: Pod state corruption, financial discrepancies, or orphaned billing sessions.  
   **Fix**:  
     - Remove the in-memory check; rely solely on the DB `UNIQUE` constraint.  
     - Handle `sqlx::Error::Database` for constraint violations explicitly:  
       ```rust
       if let Err(e) = sqlx::query(...).execute(...).await {
           if e.as_database_error().and_then(|e| e.code()) == Some("2067") { // SQLite constraint code
               return Err("Pod already has active session".into());
           }
       }
       ```
     - Wrap the entire operation (check + insert + in-memory update) in a transaction.

2. **ID: P1-002**  
   **Severity**: P1  
   **Category**: Financial Logic Flaw  
   **Location**: `routes.rs` (`refund_wallet`)  
   **Description**:  
   - Non-referenced refunds (cap: ₹500) bypass the `refunds` table audit trail. Only `wallet_transactions` is updated, violating financial audit requirements.  
   - The referenced refund path checks `already_refunded_wallet` and `already_refunded_billing` separately, allowing double-refunds if partial refunds exist across tables.  
   **Impact**: Financial loss, audit trail gaps, or double-refunds.  
   **Fix**:  
     - For non-referenced refunds, insert into `refunds` with `reference_id = NULL`.  
     - Sum refunds atomically:  
       ```rust
       let total_refunded: i64 = sqlx::query_scalar(
           "SELECT COALESCE(SUM(amount_paise), 0) FROM (
             SELECT amount_paise FROM wallet_transactions 
             WHERE reference_id=? AND txn_type IN ('refund_manual','refund_session')
             UNION ALL
             SELECT amount_paise FROM refunds WHERE billing_session_id=?
           )"
       ).bind(ref_id).bind(ref_id).fetch_one(&mut tx).await?;
       ```

3. **ID: P1-003**  
   **Severity**: P1  
   **Category**: Authentication Bypass  
   **Location**: `web/src/lib/api.ts` (`fetchApi`)  
   **Description**:  
   - The `Authorization` header is omitted if `token` is falsy, but endpoints may allow unauthenticated access. The 401 redirect only triggers for `status === 401`, ignoring 403 (Forbidden).  
   **Impact**: Unauthenticated access to privileged endpoints if misconfigured.  
   **Fix**:  
     - Explicitly reject unauthenticated requests unless endpoints are public:  
       ```typescript
       if (!token && !PUBLIC_PATHS.includes(path)) throw new Error("Unauthenticated");
       ```
     - Handle 403 similarly to 401.

#### **P2 (Functional/Correctness) Issues**

1. **ID: P2-001**  
   **Severity**: P2  
   **Category**: State Desynchronization  
   **Location**: `ws/mod.rs` (Heartbeat handler)  
   **Description**:  
   - The heartbeat updates `ip_address`, `driving_state`, and `game_state` but ignores `installed_games` and `screen_blanked`. Agents may reset these fields to default values.  
   **Impact**: Frontends show stale/incorrect game availability or screen state.  
   **Fix**: Merge agent-provided fields with server-managed fields:  
     ```rust
     existing.ip_address = pod_info.ip_address.clone();
     existing.driving_state = pod_info.driving_state;
     existing.game_state = pod_info.game_state;
     // Preserve server-managed fields:
     // existing.billing_session_id, etc.
     ```

2. **ID: P2-002**  
   **Severity**: P2  
   **Category**: Incorrect Price Calculation  
   **Location**: `billing.rs` (`start_billing_session`)  
   **Description**:  
   - `allocated_seconds` uses `split_duration_minutes` only if `split_count > 1`. If `split_count=1` and `split_duration_minutes` is provided, it falls back to `custom_duration_minutes` or tier duration, ignoring the split value.  
   **Impact**: Incorrect session duration when splits are configured.  
   **Fix**: Prioritize `split_duration_minutes` if provided, regardless of `split_count`:  
     ```rust
     let allocated_seconds = split_duration_minutes
         .or(custom_duration_minutes)
         .map(|m| m * 60)
         .unwrap_or(tier.2 as u32 * 60);
     ```

3. **ID: P2-003**  
   **Severity**: P2  
   **Category**: Missing Validation  
   **Location**: `routes.rs` (`refund_wallet`)  
   **Description**:  
   - `staff_id` is not validated for permissions. Any staff can issue refunds up to ₹5000.  
   **Impact**: Unauthorized refunds by low-privilege staff.  
   **Fix**: Enforce role-based access control (RBAC):  
     ```rust
     if !claims.0.roles.contains("finance") {
         return Err("Permission denied".into());
     }
     ```

#### **P3 (Reliability/Edge Case) Issues**

1. **ID: P3-001**  
   **Severity**: P3  
   **Category**: Poor Error Handling  
   **Location**: `wallet.rs` (`debit`)  
   **Description**:  
   - The error `"Insufficient balance"` is not contextualized (e.g., current balance vs. required amount).  
   **Impact**: User confusion during failed transactions.  
   **Fix**: Return balance details:  
     ```rust
     let balance: i64 = sqlx::query_scalar("SELECT balance_paise FROM wallets WHERE driver_id=?")
         .bind(driver_id)
         .fetch_one(conn).await?;
     if balance < amount_paise {
         return Err(format!("Insufficient balance: {} < {}", balance, amount_paise));
     }
     ```

2. **ID: P3-002**  
   **Severity**: P3  
   **Category**: Normalization Inconsistency  
   **Location**: `ws/mod.rs` (Registration)  
   **Description**:  
   - `normalize_pod_id` is used during agent registration but not in the heartbeat handler. Non-normalized IDs in heartbeats fail updates.  
   **Impact**: Pod state desynchronization if agents use non-canonical IDs.  
   **Fix**: Apply `normalize_pod_id` in the heartbeat handler:  
     ```rust
     let canonical_id = normalize_pod_id(&pod_info.id).unwrap_or(pod_info.id.clone());
     if let Some(existing) = pods.get_mut(&canonical_id) { ... }
     ```

3. **ID: P3-003**  
   **Severity**: P3  
   **Category**: Unbounded Resource Consumption  
   **Location**: `ac_launcher.rs` (`validate_content_id`)  
   **Description**:  
   - Length check (`value.len() > 128`) prevents overflow but truncation is not applied. Long IDs may break downstream systems.  
   **Impact**: Service disruption or crashes.  
   **Fix**: Truncate and log:  
     ```rust
     let value = if value.len() > 128 { &value[..128] } else { value };
     ```

4. **ID: P3-004**  
   **Severity**: P3  
   **Category**: Retry Hazard  
   **Location**: `kiosk/src/lib/api.ts` (`fetchApi`)  
   **Description**:  
   - Retries occur on network errors but not on HTTP 5xx, leading to kiosk unavailability during transient server failures.  
   **Impact**: Kiosk freezes during backend outages.  
   **Fix**: Retry on 5xx statuses:  
     ```typescript
     if (res.status >= 500 && res.status < 600 && attempt < 2) {
         await new Promise(r => setTimeout(r, 500 * Math.pow(2, attempt)));
         continue;
     }
     ```

### Summary
- **P1 Issues**: 3 (Race conditions, financial flaws, auth bypass).  
- **P2 Issues**: 3 (State desync, pricing logic, missing validation).  
- **P3 Issues**: 4 (Error handling, normalization, resource limits, retries).  

**Critical Recommendations**:  
1. Use DB transactions for all billing/wallet operations.  
2. Implement RBAC for financial endpoints.  
3. Reconcile server and agent state via startup hooks (load active sessions from DB).  
4. Add integration tests for concurrent session starts and refund workflows.
