### **P1-01: Race Condition in Billing Session Start Allows Double-Billing**  
- **Severity:** P1 (Critical)  
- **Category:** Data Integrity / Financial Risk  
- **Location:** `billing.rs` — `start_billing_session`  
- **Description:**  
  The two-phase check (in-memory read → DB insert) is **not atomic** and relies solely on a **partial unique index** for race protection. However, the **in-memory read lock is released before the DB insert**, creating a TOCTOU (Time-of-Check-Time-of-Use) window. Multiple concurrent requests can pass the in-memory check and proceed to DB insert, where only one will succeed due to the unique constraint — but **the others will fail unpredictably**, potentially causing inconsistent state or denial of service.  
  Worse: if the DB insert fails (e.g., network glitch), the session is not recorded, but the **in-memory timer is inserted anyway**, leading to a **zombie session** that blocks the pod indefinitely.

- **Impact:**  
  - **Financial loss**: Users may be charged for sessions that never started.  
  - **Denial of service**: Pod becomes unusable due to phantom active session.  
  - **Data inconsistency**: In-memory state diverges from DB.

- **Fix:**  
  Remove reliance on in-memory check as a fast path. **Use only the DB-level unique constraint as the source of truth**. Wrap the entire operation (check tier, compute, insert) in a **database transaction** and **retry on unique constraint violation** (with exponential backoff). Only after successful DB insert should in-memory state be updated.

  ```rust
  // Remove in-memory read check at start
  // Wrap everything in a retry loop on constraint violation
  let mut retries = 0;
  loop {
      let result = /* DB transaction with insert */;
      match result {
          Ok(_) => break,
          Err(e) if is_unique_violation(&e) && retries < 3 => {
              retries += 1;
              tokio::time::sleep(Duration::from_millis(50 * (2u64.pow(retries)))).await;
          }
          Err(e) => return Err(e.into()),
      }
  }
  // THEN update in-memory state
  ```

### **P1-02: Wallet Refund Bypasses Atomicity for Non-Referenced Refunds**  
- **Severity:** P1 (Critical)  
- **Category:** Financial Integrity / Data Loss  
- **Location:** `routes.rs` — `refund_wallet` (non-reference branch)  
- **Description:**  
  When `reference_id.is_none()`, the refund uses `wallet::credit()` directly, **outside of the transactional block** used for referenced refunds. This creates a **non-atomic operation**: the refund is applied via `wallet::credit()`, which itself may use a separate transaction, but there is **no audit trail linkage or atomicity with external systems**.

  Worse: `wallet::credit()` is called without any **idempotency key** or **external reference**, making it impossible to detect or prevent duplicate manual refunds.

- **Impact:**  
  - **Double-refund vulnerability**: A malicious or buggy frontend can trigger multiple manual refunds under ₹500 (bypassing reference checks).  
  - **Audit gap**: No `wallet_transaction` is guaranteed to be tied to a refund reason or staff action in a tamper-proof way.  
  - **Regulatory risk**: Non-traceable financial operations.

- **Fix:**  
  **Always use a transaction** and **insert a wallet transaction record** even for manual refunds. Do **not** bypass the audit trail.

  ```rust
  // Even for non-referenced, use a transaction
  let mut tx = conn.begin().await?;
  sqlx::query("UPDATE wallets SET balance_paise = balance_paise + ? WHERE driver_id = ?")
      .execute(&mut *tx).await?;
  sqlx::query("INSERT INTO wallet_transactions (driver_id, amount_paise, txn_type, notes, staff_id, reference_id) VALUES (?, ?, 'refund_manual', ?, ?, NULL)")
      .execute(&mut *tx).await?;
  tx.commit().await?;
  ```

  Additionally, **log all manual refunds with staff ID and timestamp** and enforce **rate limiting per staff**.

### **P1-03: In-Memory State Update Not Atomic with DB → Data Loss on Crash**  
- **Severity:** P1 (Critical)  
- **Category:** Data Loss / State Inconsistency  
- **Location:** `billing.rs` — `start_billing_session` (in-memory insert after DB)  
- **Description:**  
  The **in-memory `active_timers` and `pods` state are updated AFTER the DB insert**. If the server crashes between DB commit and in-memory update, the **DB reflects an active session**, but the **in-memory state does not**, causing:
  - Timers not running
  - Pod status not updated
  - WebSocket events not sent
  - Session may appear inactive until restart

- **Impact:**  
  - **Billing stops prematurely** (no timer → no ticks → session ends)  
  - **User charged but session cut short**  
  - **Data loss on restart**: In-memory state is lost; recovery logic not shown.

- **Fix:**  
  **Update in-memory state within the same transaction scope** (logically, not technically), or use a **two-phase commit pattern with recovery**:
  - On startup, **reconcile DB active sessions with in-memory state**.
  - Or: **Only update in-memory state if DB insert succeeds**, and **ensure all state changes are idempotent**.
  - Add **startup recovery**: query DB for all `status = 'active'` sessions and rebuild `active_timers` and `pods` state.

### **P2-01: Split Duration Logic Incorrectly Filters on `split_count` Presence**  
- **Severity:** P2 (Functional)  
- **Category:** Business Logic  
- **Location:** `billing.rs` — `allocated_seconds` calculation  
- **Description:**  
  ```rust
  if let Some(split_dur) = split_duration_minutes.filter(|_| split_count.unwrap_or(1) > 1)
  ```
  This logic **assumes `split_count` is always >1 if present**, but `split_count` could be `Some(1)`, which should **not** trigger split mode. The filter uses `split_count.unwrap_or(1)`, which defaults to 1, so `>1` fails. But if `split_count` is `Some(1)`, it still passes the `Some` check, yet `>1` is false — so split mode is **not** used, which is correct. However, the logic is **fragile and unclear**.

  Worse: if `split_count` is `None`, it defaults to 1 → `>1` false → uses custom or tier duration. But **what if split mode is intended with `split_count=None`?** The API is ambiguous.

- **Impact:**  
  - Incorrect session duration if split logic is misunderstood.
  - Frontend may pass `split_count=1` expecting split mode, but it’s ignored.

- **Fix:**  
  Clarify business rule: **split mode only if `split_count > 1` AND `split_duration_minutes.is_some()`**. Enforce explicitly:

  ```rust
  let final_split_count = split_count.unwrap_or(1);
  let allocated_seconds = if final_split_count > 1 && split_duration_minutes.is_some() {
      split_duration_minutes.unwrap() * 60
  } else {
      custom_duration_minutes.map(|m| m * 60).unwrap_or(tier.2 as u32 * 60)
  };
  ```

### **P2-02: WebSocket Heartbeat Overwrites Core-Managed Fields (Potential Conflict)**  
- **Severity:** P2 (Functional)  
- **Category:** State Management  
- **Location:** `ws/mod.rs` — `Heartbeat` handler  
- **Description:**  
  The heartbeat updates `ip_address`, `driving_state`, `game_state`, but **explicitly avoids updating core-managed fields** (`billing_session_id`, `status`, etc.). However, if the **agent sends stale or incorrect data**, it could overwrite valid state. For example:
  - Agent restarts and sends old `game_state = "idle"` while session is active.
  - `ip_address` change not validated (could be spoofed).

- **Impact:**  
  - UI shows incorrect game state.
  - Kiosk may mislead user.
  - Security: IP spoofing could mislead ops.

- **Fix:**  
  - **Validate agent identity** before accepting heartbeat.
  - **Only accept state changes that are allowed** (e.g., game_state only if session active).
  - **Add versioning or sequence numbers** to reject stale updates.
  - **Log unexpected state changes** for audit.

### **P2-03: Wallet Debit Does Not Prevent Negative `total_debited_paise`**  
- **Severity:** P2 (Functional)  
- **Category:** Data Integrity  
- **Location:** `wallet.rs` — `debit()`  
- **Description:**  
  The `total_debited_paise` is incremented by `amount_paise`, but **no check prevents overflow**. While `balance_paise` is checked for sufficiency, `total_debited_paise` is a cumulative counter and could **overflow on i64** with very large debits over time.

- **Impact:**  
  - Silent data corruption if counter overflows.
  - Analytics and reporting broken.

- **Fix:**  
  Add overflow check:
  ```rust
  .bind(amount_paise).bind(amount_paise).bind(driver_id).bind(amount_paise)
  // Add: AND total_debited_paise + ? <= i64::MAX
  ```

  Or use `DECIMAL` in DB for financial totals.

### **P3-01: Kiosk `fetchApi` Retry Logic Ignores Non-Network Errors**  
- **Severity:** P3 (Reliability)  
- **Category:** Resilience  
- **Location:** `kiosk/src/lib/api.ts` — `fetchApi`  
- **Description:**  
  Retry is only attempted on `TypeError` or `AbortError`. However, **5xx server errors** (e.g., 503) are not retried, even though they may be transient. The kiosk is in an unattended environment — transient failures should be retried.

- **Impact:**  
  - Kiosk may fail to start session due to temporary backend glitch.
  - Poor UX in high-latency environments.

- **Fix:**  
  Retry on **5xx status codes** and **timeout/network errors**:
  ```ts
  const isTransient = isNetworkError || (res && res.status >= 500);
  if (!isTransient || attempt >= 2) break;
  ```

### **P3-02: `normalize_pod_id` Failure Falls Back to Raw ID — Risk of Inconsistent Keys**  
- **Severity:** P3 (Reliability)  
- **Category:** State Consistency  
- **Location:** Multiple — `billing.rs`, `ws/mod.rs`  
- **Description:**  
  `normalize_pod_id(&pod_id).unwrap_or(pod_id)` uses raw ID on failure. If normalization fails inconsistently (e.g., due to transient error), the same pod may be registered under **different keys** (e.g., `pod-01` vs `POD-01`), causing **duplicate entries or missed updates**.

- **Impact:**  
  - Pod state split across keys.
  - Billing session not found.
  - WebSocket messages not delivered.

- **Fix:**  
  **Fail hard on normalization failure** or **ensure normalization is pure and infallible**. If fallback is needed, **log warning and use canonical form**.

  ```rust
  let canonical_id = normalize_pod_id(&pod_info.id)
      .unwrap_or_else(|e| { log::warn!("Invalid pod ID: {}, using raw", e); pod_info.id.clone() });
  ```

### **P3-03: Content ID Validation Allows Path Traversal via `...` or `.` Sequences**  
- **Severity:** P3 (Edge Case) → Could be P1 if exploited  
- **Category:** Security / Input Validation  
- **Location:** `ac_launcher.rs` — `validate_content_id`  
- **Description:**  
  While `..` is blocked, sequences like `...`, `....`, or `././` are **not blocked**. Some filesystems or parsers may interpret `...` as a directory (though rare). Also, `.` at start/end may cause issues.

- **Impact:**  
  - Potential path traversal if used in file paths later.
  - Injection into game launcher commands.

- **Fix:**  
  Add stricter rules:
  - Reject strings with `.` at start/end.
  - Reject `...`, `....`, etc.
  - Reject sequences of `.` > 1 unless part of a valid filename (e.g., `v1.2.3`).

  Or: **Whitelist only known-safe patterns** (e.g., `^[a-zA-Z0-9][a-zA-Z0-9._-]*$`).

### **P3-04: Web `fetchApi` Redirects on 401 May Cause Infinite Loop**  
- **Severity:** P3 (Reliability)  
- **Category:** UX / Browser Behavior  
- **Location:** `web/src/lib/api.ts` — `fetchApi`  
- **Description:**  
  On 401, it clears token and redirects to `/login`. If the login page also makes API calls, and token is not fully cleared, it may **trigger another 401 → redirect loop**.

- **Impact:**  
  - Infinite redirect loop.
  - User cannot log in.

- **Fix:**  
  Only redirect if not already on `/login`:
  ```ts
  if (res.status === 401 && typeof window !== "undefined") {
      clearToken();
      if (window.location.pathname !== '/login') {
          window.location.href = "/login";
      }
      throw new Error("Unauthorized");
  }
  ```
