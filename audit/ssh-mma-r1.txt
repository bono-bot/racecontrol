### Issue 1: Post-Quantum Warning
- **Root Cause**: The client (OpenSSH 10.2p1) supports post-quantum key exchange algorithms, but the servers (OpenSSH for Windows 9.5p2) do not. The client warns because the connection is not using a post-quantum algorithm.
- **Severity**: P3 (cosmetic, but annoying and affects scripting)
- **Risk**: The warning itself is not an immediate security risk, but it indicates that the connection is vulnerable to future quantum computing attacks. However, note that the current key exchange algorithms are still considered secure against classical computers. The main issue is the noise in the output.
- **Fix**: Upgrade the server-side OpenSSH to a version that supports post-quantum key exchange (if available for Windows). Alternatively, suppress the warning on the client side by using `-o LogLevel=ERROR` or by modifying the client configuration. However, upgrading the server is the proper fix.

### Issue 2: Exit code 255 on command failures
- **Root Cause**: When a command fails on the remote server (e.g., due to a missing Unix tool), the remote shell (cmd.exe) returns a non-zero exit code. However, the SSH server on Windows might be translating that into an exit code 255, which typically indicates an SSH protocol error or connection failure. This happens because the SSH server might be misreporting the exit status when the remote command fails in certain ways (especially when the command is not found). 
- **Severity**: P2 (degrades usability and complicates error handling in scripts)
- **Risk**: Misleading error codes can cause scripts to misinterpret failures (e.g., thinking the SSH connection dropped when it's just a command not found). This can lead to incorrect recovery actions and operational confusion.
- **Fix**: 
  - Ensure that the remote commands are available (e.g., by installing Unix tools on Windows, perhaps via Git for Windows or WSL). 
  - Alternatively, change the default shell to a shell that handles such errors better (like bash from Git for Windows) and set it as the default shell for SSH (via the `DefaultShell` registry key). 
  - Also, note that in cmd.exe, if a command is not found, it returns error code 9009. The SSH server should propagate that exit code. However, there might be a bug in the Windows OpenSSH server. Check the version and consider updating. If updating doesn't help, a workaround is to use a shell script that captures the error and returns the correct exit code.

### Issue 3: No keepalive configured
- **Root Cause**: The `sshd_config` file has `ClientAliveInterval` set to the default value of 0, meaning the server does not send keepalive packets. This can cause the connection to drop due to network timeouts (e.g., NAT table expiration, firewall idle timeouts).
- **Severity**: P2 (can break long-running operations)
- **Risk**: Long-running commands (like builds or large file transfers) may be interrupted unexpectedly, leading to incomplete operations and requiring manual restart.
- **Fix**: Configure `ClientAliveInterval` (e.g., 60 seconds) and `ClientAliveCountMax` (e.g., 3) in the server's `sshd_config` to send keepalive messages.

### Issue 4: Version mismatch
- **Root Cause**: The client is using OpenSSH 10.2p1 (with OpenSSL 3.5.5) and the servers are using OpenSSH for Windows 9.5p2 (with LibreSSL 3.8.2). This mismatch might cause compatibility issues, including the post-quantum warning and potentially suboptimal algorithm negotiation.
- **Severity**: P2 (could lead to security and compatibility issues)
- **Risk**: Using older server versions might expose the system to known vulnerabilities and prevent the use of newer, more secure algorithms. Also, the default shell being cmd.exe might not be suitable for commands expecting a Unix-like environment.
- **Fix**: Upgrade the server-side OpenSSH to match the client version (if possible) or at least to a version that supports the same features. Also, consider setting the `DefaultShell` registry key to a more suitable shell (like bash) for users who run Unix commands.

### Issue 5: sshd_config is almost entirely default
- **Root Cause**: The SSH server configuration is not optimized for the environment. Default settings might not be suitable for a fleet (e.g., MaxSessions, keepalive, banner suppression, and key file locations).
- **Severity**: P2 (can lead to scalability issues, security risks, and operational problems)
- **Risk**: 
  - Without tuning `MaxSessions`, the server might run out of resources under heavy load.
  - Without `ClientAliveInterval`, connections might drop (as in Issue 3).
  - Not suppressing the banner might expose version information unnecessarily (security through obscurity).
  - Using the default `AuthorizedKeysFile` locations might complicate key management, especially for administrators.
- **Fix**: 
  - Tune `MaxSessions` based on expected load.
  - Set `ClientAliveInterval` and `ClientAliveCountMax` for keepalives.
  - Set `PrintMotd no` and `PrintLastLog no` to reduce noise (if desired).
  - Use `Match` blocks to apply different settings for automation users.
  - Consider consolidating authorized keys in a single location or using group management (e.g., via Active Directory).

### Windows-Specific SSH Gotchas
1. **Default Shell**: Windows OpenSSH uses cmd.exe by default. This can be changed via the `DefaultShell` registry key (HKEY_LOCAL_MACHINE\SOFTWARE\OpenSSH). Without this, Unix commands will fail.
2. **Path Issues**: Even with a Unix shell, the PATH might not include common Unix tools (like those from Git for Windows). Ensure the PATH is set correctly in the shell's startup files.
3. **File Permissions**: The `authorized_keys` and `administrators_authorized_keys` files require strict permissions. Use `icacls` to set:
   - `administrators_authorized_keys`: owned by Administrators, with full control for SYSTEM and Administrators only.
4. **Service Account**: The SSH server runs as the SYSTEM account. This might not have the same environment variables as a user account. Use absolute paths in commands or set up the environment properly.
5. **Line Endings**: If you edit SSH configuration files or scripts on Windows, ensure they use Unix-style line endings (LF) to avoid parsing issues.
6. **Escape Characters**: In cmd.exe, the caret (^) is used as an escape character. This might interfere with complex commands. Using a Unix shell avoids this.
