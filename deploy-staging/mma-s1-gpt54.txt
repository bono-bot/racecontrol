1. Why the watchdog is not restarting `rc-agent`
   
Because `rc-sentry` is running in the wrong context.

The restart path you described depends on `session1_spawn.rs`, which uses:

- `WTSQueryUserToken`
- `CreateProcessAsUser`

That pattern is meant for a process running as **SYSTEM / service context in Session 0**, spawning a child into the interactive user session (Session 1 / Console).

Right now `rc-sentry.exe` is running in **Session 1**, started from the HKLM `Run` key. In that state:

- it is **not** acting as a service
- it likely does **not** have the privileges needed for the watchdog restart path
- the watchdog may be polling health but the actual restart branch fails or is skipped
- if the code assumes â€œservice in Session 0 spawning into active console session,â€? it may simply not work correctly when already running inside Session 1

Most likely root cause:
- `rc-sentry` watchdog restart logic was designed to run from the **RCWatchdog Windows Service**, not from a user-started copy launched via HKLM Run.
- Since there is **no RCWatchdog service installed**, you have only the interactive `rc-sentry`, and that is not sufficient to reliably relaunch `rc-agent`.

The `cmd.exe` in Session 0 is another clue that startup is currently messy:
- HKLM Run launches are racing or partially failing
- one startup path is leaving remnants in Session 0
- but nothing authoritative is managing `rc-agent`

So the answer is:

- The watchdog is not restarting `rc-agent` because `rc-sentry` is running as a **user process in Session 1**, not as the intended **service/SYSTEM launcher in Session 0**.
- The Session 1 copy is not a reliable replacement for the service-based watchdog/spawn mechanism.

2. Proper permanent fix

Yes: install and use the **RCWatchdog Windows service** on all pods, and stop relying on HKLM Run to launch both binaries.

Correct design for Windows 11 should be:

- **Windows auto-logon** ensures an interactive user session exists after boot
- **RCWatchdog service** runs as **LocalSystem** in Session 0 at boot
- service starts `rc-sentry`
- `rc-sentry` watchdog monitors `rc-agent`
- when `rc-agent` is missing/crashed, `rc-sentry` uses `WTSQueryUserToken + CreateProcessAsUser` to launch it into **Session 1**
- if `rc-agent` crashes later, watchdog restarts it the same way

What to change:

## Remove HKLM Run dependence for `rc-sentry`
Do not start `rc-sentry` from:
- `HKLM\Software\Microsoft\Windows\CurrentVersion\Run`
- startup batch files tied to user logon

That is the wrong lifetime/context for a watchdog.

## Do not depend on HKLM Run for `rc-agent`
If `rc-agent` needs to be in Session 1, it should be launched by the watchdog/service stack, not directly by Run keys.

HKLM Run is fragile because:
- it only runs after logon
- timing races with Explorer/userinit are common
- it gives you a user-context process, not a privileged launcher
- crash recovery is poor

## Install RCWatchdog as an auto-start service
Service should be:
- Startup type: **Automatic**
- Logon: **LocalSystem**
- Recovery: restart on first/second/subsequent failures

Then have that service own `rc-sentry` and the Session 1 spawning logic.

3. How auto-login should be configured

You need a guaranteed interactive session after boot, so yes, configure **real AutoAdminLogon** explicitly.

Your current state is risky:
- machines auto-logged in â€œsomehowâ€?
- `AutoAdminLogon` is not set
- behavior may break after updates, GPO changes, password changes, or boot timing changes

For Windows 11, configure these registry values:

Path:
`HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Winlogon`

Set:

- `AutoAdminLogon` = `"1"`
- `DefaultUserName` = `"user"`  or the correct local/domain username
- `DefaultDomainName` = `"SIM1"` or machine name/domain as appropriate
- `DefaultPassword` = `"<password>"`

If using a local account, `DefaultDomainName` is usually the computer name.
If using a domain-style account, set the actual domain/workgroup context correctly.

Also strongly recommended:

- `ForceAutoLogon` = `"1"`  
  Helps continue auto-logon after logoff.

If these pods are passwordless accounts today, fix that. Windows auto-logon is most reliable with:
- a known dedicated venue account
- non-expiring password
- password stored in Winlogon for auto-logon

Why this matters:
- `WTSQueryUserToken` needs an active interactive session
- no user session = nowhere to spawn `rc-agent`
- if auto-logon fails, the watchdog can run but cannot launch GUI agent properly

Also disable anything that breaks automatic console sign-in:
- â€œUsers must enter a username and password...â€? misconfiguration
- policies requiring Ctrl+Alt+Del at logon
- first-logon/privacy dialogs
- post-update blockers

Operationally, after boot you should always see:
- `explorer.exe` in Session 1 / Console
- venue user logged in
- then watchdog spawns `rc-agent` into that session
