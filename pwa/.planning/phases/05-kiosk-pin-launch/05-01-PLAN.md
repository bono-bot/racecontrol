<objective>
Build the backend POST /api/v1/kiosk/redeem-pin endpoint that validates a 6-char alphanumeric PIN against the `reservations` table, assigns the first idle pod, starts the game, and marks the PIN as redeemed.

Purpose: This is the bridge between remote booking (Phase 4) and venue game launch. Without this, a customer with a PIN has no way to start playing.

Output: Working endpoint that chains reservation lookup -> pod assignment -> billing defer -> game launch, with rate limiting and lockout protection.
</objective>

  <action>
  **In reservation.rs**, add `pub async fn redeem_pin(state: &Arc<AppState>, pin: &str) -> Result<Value, String>`:

  1. Normalize PIN: `pin.trim().to_uppercase()`. Return error if length != 6.

  2. Check for `pending_debit` status FIRST (separate query):
     ```sql
     SELECT id FROM reservations WHERE pin = ? AND status = 'pending_debit' AND expires_at > datetime('now') LIMIT 1
     ```
     If found, return `Err("Your booking is being processed. Please try again in a minute.")`.

  3. Check pod availability BEFORE consuming PIN — call `pod_reservation::find_idle_pod(state).await`. If None, return `Err("All pods are currently in use. Please wait a moment and try again.")`. Do NOT mark PIN as redeemed.

  4. Get pod_number from pods read lock: `state.pods.read().await` -> find pod by id -> `.number`.

  5. Atomic UPDATE reservations to prevent double-redeem:
     ```sql
     UPDATE reservations SET status = 'redeemed', redeemed_at = datetime('now'),
       pod_number = ?, updated_at = datetime('now')
     WHERE id = (
       SELECT id FROM reservations WHERE pin = ? AND status = 'confirmed'
         AND expires_at > datetime('now') LIMIT 1
     ) AND status = 'confirmed'
     RETURNING id, driver_id, experience_id
     ```
     If no row returned, return `Err("Invalid PIN or reservation not found")`.

  6. Get the pricing_tier_id for this reservation. The reservations table does not store pricing_tier_id directly. Look it up via the debit_intent:
     ```sql
     SELECT di.amount_paise FROM debit_intents di WHERE di.reservation_id = ? AND di.status = 'completed' LIMIT 1
     ```
     Also get the experience's linked pricing tier. Actually, the reservation only stores `experience_id`. Query `kiosk_experiences` for the experience, then use its pricing tier. Or: query the debit_intent for amount_paise, then find the matching pricing_tier by price. Simpler approach: query `kiosk_experiences` for `pricing_tier_id` field:
     ```sql
     SELECT ke.pricing_tier_id FROM kiosk_experiences ke WHERE ke.id = ?
     ```
     Bind the experience_id from the RETURNING row. If not found, use a fallback.

  7. Create pod_reservation: `pod_reservation::create_reservation(state, &driver_id, &pod_id).await?`.

  8. Create billing_session_id: `format!("deferred-{}", uuid::Uuid::new_v4())`.

  9. Call `billing::defer_billing_start(state, pod_id.clone(), driver_id.clone(), pricing_tier_id, None, None, None, None, None, None).await?`. On error, rollback: UPDATE reservations SET status = 'confirmed', redeemed_at = NULL WHERE id = ? AND status = 'redeemed'.

  10. Get driver_name: `SELECT name FROM drivers WHERE id = ?`.

  11. Get pricing tier name + duration: `SELECT name, duration_minutes FROM pricing_tiers WHERE id = ?`.

  12. Clear lock screen on pod agent:
      ```rust
      let agent_senders = state.agent_senders.read().await;
      if let Some(sender) = agent_senders.get(&pod_id) {
          let _ = sender.send(rc_common::protocol::CoreToAgentMessage::ClearLockScreen).await;
      }
      drop(agent_senders);
      ```

  13. Call `auth::launch_or_assist(state, &pod_id, &billing_session_id, &Some(experience_id), &None, &driver_name).await`.

  14. Update pod state (current_driver) and broadcast PodUpdate dashboard event — same pattern as validate_pin_kiosk lines 1418-1424.

  15. Get experience_name: `SELECT name FROM kiosk_experiences WHERE id = ?`.

  16. Return JSON:
      ```json
      {
        "pod_number": pod_number,
        "pod_id": pod_id,
        "driver_name": driver_name,
        "experience_name": experience_name,
        "tier_name": tier_name,
        "allocated_seconds": duration_minutes * 60,
        "billing_session_id": billing_session_id
      }
      ```

  <action>
  Add in-memory lockout tracking for the redeem-pin endpoint. Two layers:

  **Layer 1: Lockout tracking (in the handler)**

  1. Add a `static` or `AppState` field for lockout state. Preferred: use `once_cell::sync::Lazy` with `std::sync::Mutex<HashMap<IpAddr, PinLockoutState>>` at module level in routes.rs (avoids modifying AppState):
     ```rust
     use std::collections::HashMap;
     use std::net::{IpAddr, SocketAddr};
     use std::sync::Mutex;
     use std::time::Instant;

     struct PinLockoutState {
         fail_count: u32,
         last_attempt: Instant,
         locked_until: Option<Instant>,
     }

     static PIN_LOCKOUT: std::sync::LazyLock<Mutex<HashMap<IpAddr, PinLockoutState>>> =
         std::sync::LazyLock::new(|| Mutex::new(HashMap::new()));
     ```

  2. At the TOP of `kiosk_redeem_pin` handler, extract client IP from `ConnectInfo<SocketAddr>` extractor (add to handler signature).

  3. Check lockout FIRST:
     - Lock the mutex, get entry for this IP
     - If `locked_until` is Some and > now, return `{ "error": "Too many failed attempts. Try again in X seconds.", "locked_until_seconds": remaining }` with HTTP 429
     - If `locked_until` is Some but expired, reset the entry

  4. On FAILED PIN validation (after calling reservation::redeem_pin and getting Err):
     - Increment fail_count for this IP
     - Update last_attempt
     - If fail_count >= 10, set `locked_until = Instant::now() + Duration::from_secs(300)` (5 min cooldown)
     - Include `remaining_attempts: 10 - fail_count` in error response (but only if under 10)

  5. On SUCCESSFUL validation: reset the lockout entry for this IP (clear fail_count).

  6. Cleanup: every 100th request (or when map size > 1000), prune entries older than 10 minutes to prevent memory growth.

  **Layer 2: Tower-governor rate limit (route-level)**

  Check if the existing `auth_rate_limit_layer()` from `auth/rate_limit.rs` can be applied to just the redeem-pin route. If it uses tower-governor with 5/min per IP, apply it:
  ```rust
  .route("/kiosk/redeem-pin", post(kiosk_redeem_pin))
  .layer(auth_rate_limit_layer())
  ```
  If `auth_rate_limit_layer` returns a middleware layer, wrap just this route. If it's not easily composable, skip the tower-governor layer — the in-handler lockout is sufficient for defense-in-depth.

  **Return format for locked state:**
  ```json
  { "error": "Too many failed attempts. Please wait 4 minutes and 32 seconds.", "lockout_remaining_seconds": 272 }
  ```

  **Return format for failed but not locked:**
  ```json
  { "error": "Invalid PIN or reservation not found", "remaining_attempts": 7 }
  ```
