The script must perform TWO gates per pod:

**Gate 1: Model presence** — Verify rp-debug model is installed
- Execute via :8090/exec: `curl -s http://127.0.0.1:11434/api/tags`
- Parse the JSON response to check if "rp-debug" appears in the model list
- PASS if found, FAIL if not found

**Gate 2: Model response time** — Verify rp-debug responds <5s
- Execute via :8090/exec: `curl -s --max-time 8 -X POST http://127.0.0.1:11434/api/generate -d "{\"model\":\"rp-debug\",\"prompt\":\"diagnose: test ping\",\"stream\":false}"`
- The exec timeout_ms should be 10000 (10s) to give curl 8s + overhead
- Parse the response to verify a non-empty "response" field exists
- Measure wall clock time — PASS if the response arrives in <5s, FAIL if >5s or empty
- Note: First invocation after model load may be slower. The 8s curl timeout accommodates cold start. The <5s check is on the total exec round-trip, not just LLM inference.

For Gate 1, the /exec command should be:
`curl -s http://127.0.0.1:11434/api/tags`
Parse stdout from exec response for "rp-debug".

For Gate 2, the /exec command should be:
`powershell -NoProfile -Command "$sw = [Diagnostics.Stopwatch]::StartNew(); $r = curl.exe -s --max-time 8 -X POST http://127.0.0.1:11434/api/generate -d '{\\\"model\\\":\\\"rp-debug\\\",\\\"prompt\\\":\\\"diagnose: test ping\\\",\\\"stream\\\":false}'; $sw.Stop(); Write-Output \"TIME_MS=$($sw.ElapsedMilliseconds)|RESP=$r\""`
This gives both timing and response in one exec call. Parse TIME_MS for <5000 check, RESP for non-empty response field.

If powershell timing is too fragile, use a simpler approach: just run the curl and check if the exec response arrives (the /exec endpoint has its own timeout). Use timeout_ms: 8000 on the /exec call. If it returns within the timeout and has a non-empty response, pass.

Use the simpler approach: exec with timeout_ms=8000. If the exec completes and stdout contains "response", pass. The 8s timeout IS the <5s gate (with margin for network overhead from James machine to pod).

Actually, to properly measure <5s, use `date +%s%N` before and after the curl to the exec endpoint from James's machine. The full round-trip (James -> pod :8090 -> pod curl localhost:11434 -> response back) in <5s proves the LLM is responsive.

Final approach (simple, matches close-wait.sh style):
```bash
START_MS=$(date +%s%3N)
EXEC_RESP=$(curl -s --connect-timeout 3 --max-time 10 \
    -X POST "http://${POD_IP}:8090/exec" \
    -H "Content-Type: application/json" \
    -d '{"cmd":"curl -s --max-time 8 -X POST http://127.0.0.1:11434/api/generate -d \"{\\\"model\\\":\\\"rp-debug\\\",\\\"prompt\\\":\\\"diagnose: test ping\\\",\\\"stream\\\":false}\"","timeout_ms":10000}' \
    2>/dev/null)
END_MS=$(date +%s%3N)
ELAPSED_MS=$((END_MS - START_MS))
```
Check ELAPSED_MS < 5000 AND stdout contains "response".
