<objective>
Wire the immediate fix application path for Tier 1-3 with a 30-second post-fix verification loop. No human approval queue for these tiers. Verification confirms the fix worked or escalates.

Purpose: Close the loop from "fix applied" to "fix confirmed working." Without verification, the system assumes success and moves on while the problem persists. The 30-second window catches regressions immediately.

Output: verify_fix() function in tier_engine, wired after every Tier 1-3 fix, emitting outcome FleetEvents.
</objective>

1. Add `verify_fix()` async function to tier_engine.rs:

```rust
/// Verify that a fix actually resolved the issue.
/// Waits up to 30 seconds, checking every 5 seconds (6 attempts).
/// Returns true if the specific anomaly condition has cleared.
async fn verify_fix(
    trigger: &DiagnosticTrigger,
    failure_monitor_rx: &tokio::sync::watch::Receiver<crate::failure_monitor::FailureMonitorState>,
) -> bool
```

Implementation per trigger type:
- `ProcessCrash { process_name }` -> check sysinfo: process_name not in WerFault state. Use `tokio::task::spawn_blocking` for sysinfo calls.
- `GameLaunchFail` -> check failure_monitor state: `game_pid.is_some()`
- `DisplayMismatch` -> check failure_monitor state: `edge_process_count > 0`
- `WsDisconnect` -> check: ws_connected (from failure_monitor or heartbeat)
- `SentinelUnexpected { file_name }` -> check: `!std::path::Path::new(&format!("C:\\RacingPoint\\{}", file_name)).exists()` (path traversal guard: reject if file_name contains `..` or `\`)
- `ErrorSpike` -> check failure_monitor: error rate back below threshold
- `ViolationSpike` -> check: violation delta stabilized
- `Periodic` -> always return true (periodic scans are informational)
- `HealthCheckFail` -> check: self HTTP health responds
- `PreFlightFailed` -> check: re-run the specific pre-flight check (or return true if not easily re-checkable)
- POS-specific triggers -> check relevant POS state from failure_monitor

The function loops up to 6 times with 5-second sleep between checks (total 30s). If any check returns true (fixed), return immediately. If all 6 attempts fail, return false.

2. Wire `verify_fix()` into the main loop in `run_supervised()`:

After `run_tiers()` returns `TierResult::Fixed { tier, action }` for tier 1-3:
- If tier <= 3 (auto-fix tiers): call `verify_fix(&event.trigger, &failure_monitor_rx).await`
- Pass the broadcast_tx (from Plan 01) to be able to emit FleetEvents
- If verified (returns true):
  - Log INFO: "Fix verified: tier={tier} action={action} trigger={:?}"
  - Send `FleetEvent::FixApplied { node_id, tier, action, trigger, timestamp }`
  - Update KB recording (from Plan 03) with verification_result="verified_pass"
- If NOT verified (returns false):
  - Log WARN: "Fix verification FAILED: tier={tier} action={action} — escalating"
  - Send `FleetEvent::FixFailed { node_id, tier, reason: "verification_failed", trigger, timestamp }`
  - Update KB recording with verification_result="verified_fail"
  - Escalate: re-emit the event but with a flag to skip tiers <= current tier (so next run tries tier+1)
  - For Tier 3 failure: emit `FleetEvent::Escalated` (Tier 4/5 handle separately)

3. For Tier 4/5 results: verification is different (model-suggested fixes may need longer observation). For now, log but don't verify within 30s. Mark verification_result="not_verified" in KB.

4. Add `broadcast_tx` and `failure_monitor_rx` access to the verification path. The `failure_monitor_rx` is already passed to `run_supervised()`. The `broadcast_tx` needs to be passed from main.rs (add parameter to spawn/run_supervised if not already there from Plan 01).

Standing rules: no .unwrap(), no lock across await, path traversal guard on sentinel checks, use spawn_blocking for sysinfo.
