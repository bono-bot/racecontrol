**knowledge_base.rs changes:**

1. Add `record_resolution()` public method to KnowledgeBase:
```rust
pub fn record_resolution(
    &self,
    problem_key: &str,
    problem_hash: &str,
    symptoms: &str,
    fix_action: &str,
    fix_type: &str,       // "deterministic", "kb_cached", "model_suggested"
    tier: u8,
    verification_result: &str,  // "verified_pass", "verified_fail", "not_verified"
    node_id: &str,
    diagnosis_method: Option<&str>,
) -> anyhow::Result<()>
```
This should:
- Check if a solution with the same `problem_hash` already exists
- If YES: increment `success_count` (if verification_result is "verified_pass") or `fail_count`, update `updated_at`, update `confidence` based on new success/fail ratio
- If NO: INSERT new row with uuid, `confidence = 1.0` for verified_pass or `0.5` for others, `source_node = node_id`, current timestamp for `created_at` and `updated_at`
- Log at INFO level: "KB recorded: tier={tier} action={fix_action} hash={problem_hash} verification={verification_result}"
- Use `?` for error handling, never `.unwrap()`

2. Add `lookup_by_hash()` public method (may already exist — check and enhance):
```rust
pub fn lookup_by_hash(&self, problem_hash: &str) -> anyhow::Result<Option<Solution>>
```
Returns the highest-confidence solution for this problem hash, filtered to `confidence >= HIGH_CONFIDENCE_THRESHOLD`.

**tier_engine.rs changes:**

1. Add `kb: Arc<std::sync::Mutex<KnowledgeBase>>` parameter to `spawn()` and `run_supervised()`. In main.rs, wrap the existing KnowledgeBase instance in `Arc<Mutex<>>` and pass it. Use `std::sync::Mutex` (NOT tokio::sync::Mutex) since KB ops are synchronous SQLite. Access pattern: `let kb_guard = kb.lock().map_err(|e| ...)?;` in a tight scope, drop before any .await.

2. **KB-first gate BEFORE Tier 3/4:** In `run_tiers()`, after Tier 2 runs (and before Tier 3):
   - Generate `problem_hash` from the event trigger (use existing `make_dedup_key()` or similar)
   - Call `kb.lock()` in tight scope, then `kb_guard.lookup_by_hash(&problem_hash)`
   - If a high-confidence solution is found:
     - Log: "KB hit: applying cached solution (confidence={}, tier=2.5, $0 cost)"
     - Apply the cached fix_action (call the appropriate Tier 1/2 executor)
     - Record the application: `kb_guard.record_resolution(...)` with fix_type="kb_cached"
     - Send `FleetEvent::FixApplied` via broadcast (from Plan 01)
     - Return `TierResult::Fixed { tier: 2, action: format!("KB: {}", solution.fix_action) }`
     - This SKIPS Tier 3/4 entirely — $0 cost
   - If no KB hit: proceed to Tier 3/4 as normal

3. **Record after EVERY TierResult::Fixed:** After `run_tiers()` returns in the main loop, if result is `TierResult::Fixed`:
   - Generate problem_hash from event trigger
   - Call `record_resolution()` with the tier, action, and "verified_pass" (verification comes in Plan 04)
   - This captures Tier 1 deterministic fixes too — they become future Tier 2 KB hits

4. **Record after TierResult::FailedToFix** too — with verification_result "verified_fail" so the KB tracks what DOESN'T work and can avoid repeating failed fixes.

Standing rules: no .unwrap(), no lock across .await (lock KB in tight scope, clone result, drop guard, then await), lifecycle logging.
