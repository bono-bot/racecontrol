**1. Add EventsQuery struct** (near other query structs in routes.rs):
```rust
#[derive(Deserialize)]
struct EventsQuery {
    event_type: Option<String>,
    pod: Option<String>,
    from: Option<String>,   // YYYY-MM-DD
    to: Option<String>,     // YYYY-MM-DD
    limit: Option<i64>,
}
```

**2. Add get_events handler** using BillingListQuery dynamic WHERE builder pattern:
- Base query: `SELECT id, event_type, source, pod, timestamp, payload FROM system_events WHERE 1=1`
- For `event_type` filter: validate chars (ascii_alphanumeric + `_` + `.`), append `AND event_type = ?`
- For `pod` filter: validate chars (ascii_alphanumeric + `-` + `_`), append `AND pod = ?`
- For `from` filter: validate format (10 chars, digits + `-`), append `AND date(timestamp) >= ?`
- For `to` filter: validate format (10 chars, digits + `-`), append `AND date(timestamp) <= ?`
- Limit: `q.limit.unwrap_or(200).min(1000)`
- ORDER BY timestamp DESC LIMIT ?
- For each row, parse payload TEXT back to serde_json::Value with `serde_json::from_str(&payload).unwrap_or(Value::String(payload.clone()))` to avoid double-encoding (Pitfall 5)
- Return `Json(json!({ "events": [...], "count": N }))`

**3. Register route in staff_routes():**
Add `.route("/events", get(get_events))` in the staff_routes function. Place it near the debug routes section, with a comment `// Phase 302: Event archive query API`.

**4. Bind values dynamically:**
Since sqlx doesn't support dynamic bind count easily, use the `sqlx::query` with manual bind chaining. Build the query string, then bind each value in order. Use `sqlx::query_as::<_, (String, String, String, Option<String>, String, String)>` for the 6-column SELECT.

Alternatively, use `sqlx::query()` and fetch rows manually with `row.get::<String, _>(0)` etc. Follow whichever pattern is used in the existing BillingListQuery handler.

**Standing rules:**
- No `.unwrap()` in handler code
- Validate ALL filter inputs with character allowlists before SQL injection
- Route in staff_routes() (NOT public_routes) — events are internal operational data
