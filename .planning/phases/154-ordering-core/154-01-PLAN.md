```rust
pub async fn place_cafe_order(
    State(state): State<Arc<AppState>>,
    Json(req): Json<PlaceOrderRequest>,
) -> Result<Json<serde_json::Value>, StatusCode> {
```

Validation (before transaction):
- `req.items` must not be empty, return 400
- Each `quantity` must be >= 1, return 400
- `driver_id` must not be empty, return 400

**CRITICAL: The atomic transaction flow.**

Because `wallet::debit()` uses the pool directly (not a transaction handle), we CANNOT include the wallet debit inside a raw SQL transaction. Instead, use this safe pattern:

Step A: Acquire a raw SQLite connection from the pool and BEGIN IMMEDIATE (exclusive write lock):
```rust
let mut conn = state.db.acquire().await.map_err(|e| { ... })?;
sqlx::query("BEGIN IMMEDIATE").execute(&mut *conn).await.map_err(|e| { ... })?;
```

Step B: Validate all items exist, are available, and check stock. Query each item:
```sql
SELECT id, name, selling_price_paise, is_countable, stock_quantity, is_available
FROM cafe_items WHERE id = ?
```
- If any item not found or not available: ROLLBACK and return 400 with `{"error": "Item not found or unavailable: {name}"}`
- If any countable item has `stock_quantity < requested_quantity`: ROLLBACK and return 400 with `{"error": "Out of stock: {name} (available: {stock}, requested: {qty})"}`

Step C: Calculate `total_paise` = sum of (selling_price_paise * quantity) for all items. Build `Vec<OrderItemDetail>`.

Step D: Decrement stock for each countable item within the transaction:
```sql
UPDATE cafe_items SET stock_quantity = stock_quantity - ?, updated_at = datetime('now')
WHERE id = ? AND is_countable = 1 AND stock_quantity >= ?
```
Check rows_affected == 1 for each. If 0, another concurrent order got the last unit — ROLLBACK and return 409 with `{"error": "Stock changed during order, please retry"}`.

Step E: Generate receipt number within the transaction:
```sql
SELECT COUNT(*) FROM cafe_orders WHERE receipt_number LIKE 'RP-' || strftime('%Y%m%d', 'now') || '-%'
```
Then format: `RP-{YYYYMMDD}-{count+1:04}` (zero-padded to 4 digits).

Step F: Generate order_id with `Uuid::new_v4().to_string()`.

Step G: COMMIT the transaction (stock is decremented, receipt reserved).

Step H: **After commit**, call `wallet::debit()` — this has its own atomic UPDATE WHERE balance >= ? RETURNING:
```rust
let (new_balance, wallet_txn_id) = wallet::debit(
    &state, &req.driver_id, total_paise,
    "cafe_order", Some(&order_id), Some(&format!("Cafe order {}", receipt_number)),
).await.map_err(|e| {
    // Wallet debit failed — we need to REVERSE the stock decrement
    // Spawn a rollback task (best-effort stock restore)
    let state_clone = state.clone();
    let items_clone = verified_items.clone(); // items with quantities
    tokio::spawn(async move {
        for item in &items_clone {
            if item.is_countable {
                let _ = sqlx::query("UPDATE cafe_items SET stock_quantity = stock_quantity + ? WHERE id = ?")
                    .bind(item.quantity).bind(&item.item_id)
                    .execute(&state_clone.db).await;
            }
        }
        tracing::warn!("Rolled back stock for failed wallet debit on order {}", order_id_for_log);
    });
    StatusCode::BAD_REQUEST
})?;
```
Return the wallet error message in JSON: `{"error": "Insufficient balance: ..."}`. Note: Map the String error to a JSON response, not just StatusCode. Use a match or `if let Err(e)` pattern.

Step I: Insert the order record:
```sql
INSERT INTO cafe_orders (id, receipt_number, driver_id, items, total_paise, wallet_txn_id, status)
VALUES (?, ?, ?, ?, ?, ?, 'confirmed')
```
The `items` field is `serde_json::to_string(&order_item_details)`.

Step J: Fire low-stock alerts (non-blocking, after insert):
```rust

    if item.is_countable {
        cafe_alerts::check_low_stock_alerts(&state.db, &state.config, &item.item_id).await;

}
```

Step K: Log and return `PlaceOrderResponse` as JSON:

**IMPORTANT:** Do NOT use `.unwrap()` anywhere. All errors must use `?` or `map_err`. Use `tracing::warn!` for error logging.

**IMPORTANT:** The `wallet::debit` call happens OUTSIDE the raw SQLite transaction because it uses `state.db` pool internally. The stock decrement is protected by the BEGIN IMMEDIATE lock. If wallet debit fails after stock commit, we rollback stock via a compensating update (Step H).
