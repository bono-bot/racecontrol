Script structure (8 gates):

**Header:**
- `#!/bin/bash` + `set -uo pipefail`
- `BASE_URL="${RC_BASE_URL:-http://192.168.31.23:8080/api/v1}"`
- `POD_ID="${TEST_POD_ID:-pod-8}"` (canary pod)

- `POD_IP=$(pod_ip "${POD_ID}")`
- `RESULTS_DIR="${RESULTS_DIR:-$(dirname "$SCRIPT_DIR")/results}"` for AI debugger log output
- `AI_LOG="${RESULTS_DIR}/ai-debugger-input.log"` (DEPL-04 target file)
- Define helper `log_to_ai_debugger()` that appends timestamped failure details to `$AI_LOG`. Args: gate_name, failure_message. Format: `[YYYY-MM-DD HH:MM:SS] GATE: $gate_name | FAILURE: $failure_message`. Create `$RESULTS_DIR` directory if it does not exist (`mkdir -p`).

**Gate 0: Server Health (pre-check)**
- `curl -s --max-time 5 "${BASE_URL}/health"` -- if unreachable, fail + `log_to_ai_debugger "server_health" "racecontrol unreachable at ${BASE_URL}"` + exit 1

**Gate 1: rc-sentry reachable on canary pod (DEPL-01)**
- `curl -s --max-time 5 "http://${POD_IP}:8091/ping"` -- pass if HTTP 200, fail + log_to_ai_debugger if not
- This confirms the deploy verification channel is available independent of rc-agent

**Gate 2: Binary size check (DEPL-01)**
- Via rc-sentry: `POST http://${POD_IP}:8091/exec` with `{"cmd": "powershell -NonInteractive -Command \"(Get-Item 'C:/RacingPoint/rc-agent.exe').Length\""}`
- Parse response for numeric file size. Store in `BINARY_SIZE`.
- pass if size > 0 (non-empty binary), fail + log_to_ai_debugger if size is 0 or command failed
- info "Binary size: ${BINARY_SIZE} bytes"

**Gate 3: Port conflict detection - kiosk :3300 (DEPL-01)**
- Via rc-sentry on SERVER (.23): Since verify.sh runs from James (.27) and needs to check server ports, use: `curl -s --max-time 5 -o /dev/null -w "%{http_code}" "http://192.168.31.23:3300/"` directly
- If HTTP code is 200, 307, or 308: pass "Kiosk :3300 is serving"
- If HTTP code is 000 (refused/timeout): info "Kiosk :3300 not responding -- checking if port is stuck"
  - Poll up to 30s (6 attempts x 5s) with `curl` waiting for :3300 to either respond or be confirmed free. The goal is detecting EADDRINUSE -- if a process holds :3300 but is not serving HTTP, the curl will return 000 but the port is still bound.
  - Use rc-sentry on server IP (192.168.31.23:8091) if available, or fall back to direct curl probing
- If after 30s still no response: fail "Kiosk :3300 not serving after 30s wait" + log_to_ai_debugger

**Gate 4: racecontrol :8080 serving (DEPL-01)**
- `curl -s --max-time 5 "${BASE_URL}/health"` -- already checked in Gate 0 but this gate specifically confirms after any simulated restart
- Check response contains valid JSON or expected health string
- pass/fail accordingly

**Gate 5: Fleet health - all 8 pods WS connected (DEPL-02)**
- `curl -s --max-time 10 "${BASE_URL}/fleet/health"` -- parse JSON with python3
- For each pod 1-8: check `ws_connected` field is true
- Count connected vs total. Pass if all 8 connected. If < 8: info which pods are disconnected, fail "Only N/8 pods connected" + log_to_ai_debugger with list of disconnected pods

**Gate 6: build_id consistency across fleet (DEPL-02)**
- From the same fleet/health response (reuse from Gate 5): extract `build_id` for each connected pod
- Collect unique build_ids. If exactly 1 unique value: pass "All pods on build ${BUILD_ID}". If > 1: fail "Build ID mismatch" + log_to_ai_debugger with per-pod build_id list

**Gate 7: installed_games validation (DEPL-02)**
- From fleet/health response: extract `installed_games` for pod-8 (canary)
- Check that the list is non-empty. Optionally verify it contains at least `assetto_corsa` (the baseline game).
- pass if non-empty, skip if field not present (older agent version), fail if empty list

**DEPL-04 integration:**
- Every `fail()` call is paired with a `log_to_ai_debugger()` call providing gate name + error details
- At end of script, if `$AI_LOG` file exists and has content, print: `info "AI debugger input written to: ${AI_LOG}"`
- The log file format is simple text, one failure per line, suitable for the AI debugger to parse
