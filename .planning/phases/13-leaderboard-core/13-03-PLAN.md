  <implementation>
    1. Restructure the track record section in persist_lap():
       a. STEP 1 — Fetch current record holder with email BEFORE upsert:
          ```sql
          SELECT tr.best_lap_ms, d.name, d.email
          FROM track_records tr
          JOIN drivers d ON tr.driver_id = d.id
          WHERE tr.track = ? AND tr.car = ?
          ```
          Store as Option<(i64, String, Option<String>)>.
       b. Determine is_record: new time < existing best (or no existing = first record).
       c. STEP 2 — Also fetch the NEW record holder's display name BEFORE the upsert (NTF-02 requirement):
          ```sql
          SELECT CASE WHEN show_nickname_on_leaderboard = 1 AND nickname IS NOT NULL
                      THEN nickname ELSE name END
          FROM drivers WHERE id = ?
          ```
          Bind lap.driver_id. Store as new_holder_name: String.
       d. STEP 3 — Execute the existing UPSERT (unchanged).
       e. STEP 4 — If is_record AND prev_record exists AND prev_holder has email:
          - Build subject: "Your {car} record at {track} has been beaten!"
          - Build body: "Hi {prev_name},\n\nYour track record at {track} in the {car} has been broken.\n\nNew record set by: {new_holder_name}\nOld time: {old_display}\nNew time: {new_display}\n\nCome back and reclaim it!\n\nhttps://app.racingpoint.cloud/leaderboard/public"

          - Clone necessary data for move into async block.
          - tokio::spawn fire-and-forget: Command::new("node").arg(&email_script).arg(&email).arg(&subject).arg(&body).kill_on_drop(true).output().await
       f. If prev_holder has no email: log at debug level, skip notification.
    2. Access email_script_path via state.config.watchdog.email_script_path (same path as email_alerts.rs).
    3. Test approach: Tests verify the data ordering by checking the DB state.
