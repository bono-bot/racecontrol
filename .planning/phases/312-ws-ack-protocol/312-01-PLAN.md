<objective>
Add confirmed-delivery ACK protocol to game launch and stop commands. Server waits up to 5 seconds for agent acknowledgment before returning success/failure to the API caller.

Purpose: Currently /games/launch and /games/stop return ok:true before the agent even receives the command. If the WS drops between queue and delivery, the command is silently lost. This phase makes command delivery confirmed.

Output: Modified protocol.rs (new CommandAck variant), state.rs (pending ack map), game_launcher.rs (wait-for-ack logic), ws/mod.rs (ack handler), ws_handler.rs (agent sends ack).
</objective>

**CoreMessage wrapper already has command_id** (DEPLOY-05):
```rust
pub struct CoreMessage {
    #[serde(default, skip_serializing_if = "Option::is_none")]
    pub command_id: Option<String>,
    #[serde(flatten)]
    pub inner: CoreToAgentMessage,
}
impl CoreMessage {
    pub fn wrap(msg: CoreToAgentMessage) -> Self {
        Self { command_id: Some(uuid::Uuid::new_v4().to_string()), inner: msg }
    }
}
```

**Agent ws_handler** receives `CoreToAgentMessage::LaunchGame` and `CoreToAgentMessage::StopGame` in a match arm. The outer handler receives `CoreMessage` which has `command_id`.

**A. game_launcher.rs — launch_game: wait for ACK after send (WSCMD-01, WSCMD-03)**

After the existing send loop (lines ~436-459) succeeds (`send_ok = true`), instead of immediately proceeding, add ACK waiting:

1. Generate a command_id: `let command_id = uuid::Uuid::new_v4().to_string();`
2. IMPORTANT: The `launch_msg` is currently a bare `CoreToAgentMessage`. The agent_senders channel type is `mpsc::Sender<CoreToAgentMessage>`. The `CoreMessage` wrapper is used at the WS serialization layer. Look at how the WS write loop in `ws/mod.rs` serializes outgoing messages — it likely wraps with `CoreMessage::wrap()`. If the channel sends `CoreToAgentMessage` directly and the WS loop wraps it, then the command_id must be injected at the WS layer.
   
   **Alternative approach (simpler):** Instead of changing the channel type, register the oneshot BEFORE sending and use the existing `CoreMessage::wrap()` at the WS serialization point. BUT — the command_id is generated at wrap time, and the game_launcher needs to know it to register the waiter.

   **Recommended approach:** Change the send flow to:
   - Create a `(oneshot::Sender, oneshot::Receiver)` pair
   - Generate command_id
   - Insert into `state.pending_command_acks` with the command_id
   - The WS write loop already wraps messages with `CoreMessage::wrap()` which generates a NEW command_id. We need to pass OUR command_id through.
   
   **Simplest correct approach:** After the existing send succeeds, register the ack waiter using the pod_id + a fresh UUID. The agent will extract the command_id from the CoreMessage wrapper it receives and echo it back. BUT — we don't control what command_id CoreMessage::wrap() generates.

   **Cleanest approach:** Change `agent_senders` channel to send `CoreMessage` instead of `CoreToAgentMessage`, so the caller controls the command_id. BUT this is a large change touching many senders.

   **ACTUAL cleanest approach (follow ExecResult pattern):** 
   - The ExecResult pattern uses `request_id` embedded IN the `CoreToAgentMessage::Exec` variant itself, not in the CoreMessage wrapper.
   - Similarly, DON'T rely on the CoreMessage wrapper command_id for ACK correlation.
   - Instead, add a `#[serde(default)] command_id: Option<String>` field to `CoreToAgentMessage::LaunchGame` and `CoreToAgentMessage::StopGame`.
   - `serde(default)` makes it backward-compatible — old servers that don't send it produce `None` on the agent.
   - The agent reads the command_id from the inner message and sends it back in CommandAck.
   - The server sets command_id before sending and registers the oneshot waiter with that id.

   Implementation:
   a. In protocol.rs, add `#[serde(default)] command_id: Option<String>` to `LaunchGame` variant (after duration_minutes) and to `StopGame` (convert from unit variant to struct: `StopGame { #[serde(default)] command_id: Option<String> }`).
   b. Update ALL places that construct `LaunchGame` (4 launcher impls in game_launcher.rs: `make_launch_message`) to accept and pass command_id.
   c. Update `StopGame` construction everywhere.

   In launch_game():
   ```rust
   let command_id = uuid::Uuid::new_v4().to_string();
   // ... set command_id on launch_msg ...
   
   // Register ACK waiter BEFORE sending
   let (ack_tx, ack_rx) = tokio::sync::oneshot::channel();
   state.pending_command_acks.write().await.insert(command_id.clone(), ack_tx);
   
   // ... existing send logic ...
   
   // After send_ok = true, wait for ACK with 5s timeout
   match tokio::time::timeout(std::time::Duration::from_secs(5), ack_rx).await {
       Ok(Ok(result)) => {
           if !result.success {
               // Agent reported failure
               tracing::warn!("Agent ACK failure for launch on pod {}: {:?}", pod_id, result.error);
               // Update tracker to Error state
               // ... (similar to the no-agent-connected error path)
               return Err(format!("Agent failed to launch on pod {}: {}", pod_id, result.error.unwrap_or_default()));
           }
           tracing::info!("Launch ACK received for pod {} (command_id={})", pod_id, command_id);
       }
       Ok(Err(_)) => {
           // Oneshot dropped — agent disconnected
           tracing::warn!("ACK channel dropped for launch on pod {} (agent disconnected?)", pod_id);
           return Err(format!("Agent disconnected before acknowledging launch on pod {}", pod_id));
       }
       Err(_) => {
           // Timeout — WSCMD-03: old agent or WS drop
           tracing::warn!("WSCMD-03: No ACK within 5s for launch on pod {} (command_id={})", pod_id, command_id);
           // Clean up pending entry
           state.pending_command_acks.write().await.remove(&command_id);
           return Err(format!("Agent did not acknowledge launch command within 5s (pod {})", pod_id));
       }
   }
   ```

AgentMessage::CommandAck { command_id, success, error } => {
