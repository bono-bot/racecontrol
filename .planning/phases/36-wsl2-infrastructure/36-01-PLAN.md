This task sets up the entire WSL2 + Salt master stack in sequence. Every command runs from James's machine (.27).

**Step 1 — Create .wslconfig for mirrored networking (BEFORE installing Ubuntu)**

Write `C:\Users\bono\.wslconfig` with exactly this content:
```ini
[wsl2]
networkingMode=mirrored
memory=4GB
swap=2GB
```
No CPU limit per user decision. Swap at 2GB (half of RAM limit, covers Salt burst startup).

**Step 2 — Install Ubuntu 24.04 in WSL2**

Run in PowerShell (no admin needed for install):
```
wsl --install -d Ubuntu-24.04 --no-launch
```
Then launch it once to complete first-run setup:
```
wsl -d Ubuntu-24.04 -- echo "first-run complete"
```
If prompted for username/password, create user `bono` with a password. This is the WSL2 Linux user.

After first-run, shut down and restart WSL2 to apply mirrored networking:
```
wsl --shutdown
```
Wait 3 seconds, then verify mirrored mode:
```
wsl -d Ubuntu-24.04 -- hostname -I
```
The output MUST contain `192.168.31.27`. If it shows `172.x.x.x`, mirrored mode is NOT active — re-check `.wslconfig` spelling and restart WSL2 again.

**Step 3 — Configure /etc/wsl.conf inside Ubuntu**

Run inside WSL2:
```bash
wsl -d Ubuntu-24.04 -e bash -c 'sudo tee /etc/wsl.conf << '"'"'WSLCONF'"'"'
[boot]
systemd=true
command="service salt-master start && service salt-api start"
WSLCONF'
```

**Step 4 — Install salt-master 3008 LTS via bootstrap script**

Run inside WSL2:
```bash
wsl -d Ubuntu-24.04 -e bash -c '
curl -fsSL https://github.com/saltstack/salt-bootstrap/releases/latest/download/bootstrap-salt.sh -o /tmp/bootstrap-salt.sh
sudo sh /tmp/bootstrap-salt.sh -M -N -P stable 3008
'
```
Flags: -M = install master, -N = no minion on this machine, -P = allow pip if needed, stable 3008 = 3008 LTS.

After bootstrap, explicitly install salt-api (bootstrap may not include it):
```bash
wsl -d Ubuntu-24.04 -e bash -c 'sudo apt-get install -y salt-api'
```

Verify versions:
```bash
wsl -d Ubuntu-24.04 -e bash -c 'salt-master --version && salt-api --version'
```
Both must show `3008.x`.

**Step 5 — Write /etc/salt/master config**

Run inside WSL2:
```bash
wsl -d Ubuntu-24.04 -e bash -c 'sudo tee /etc/salt/master << '"'"'SALTCONF'"'"'
# Salt master config for Racing Point venue LAN
# Phase 36 — WSL2 Infrastructure

# Bind all interfaces — mirrored mode picks up 192.168.31.27
interface: 0.0.0.0

# Manual key acceptance (closed venue LAN)
auto_accept: False

# File roots — rc-agent.exe and state files will live here (Phase 38+)
file_roots:
  base:
    - /srv/salt

pillar_roots:
  base:
    - /srv/pillar

  port: 8000
  disable_ssl: True

# External auth for salt-api — PAM uses Linux system users
external_auth:
  pam:
    saltadmin:
      - ".*"
      - "@wheel"
      - "@runner"

# Faster dead-minion detection (Phase 37+)
ping_interval: 20
SALTCONF'
```

Create the required directories:
```bash
wsl -d Ubuntu-24.04 -e bash -c 'sudo mkdir -p /srv/salt /srv/pillar'
```

**Step 6 — Create saltadmin PAM user for salt-api auth**

Run inside WSL2:
```bash
wsl -d Ubuntu-24.04 -e bash -c '
sudo useradd -m -s /bin/bash saltadmin
echo "saltadmin:RacingPoint2026Salt!" | sudo chpasswd
'
```
Password: `RacingPoint2026Salt!` — this will go into racecontrol.toml `[salt]` section in Phase 38.

**Step 7 — Configure sudoers for passwordless service start**

This is CRITICAL for Task Scheduler auto-start (INFRA-05) to work without password prompts:
```bash
wsl -d Ubuntu-24.04 -e bash -c '
echo "bono ALL=(ALL) NOPASSWD: /usr/sbin/service salt-master *, /usr/sbin/service salt-api *" | sudo tee /etc/sudoers.d/salt-services
sudo chmod 0440 /etc/sudoers.d/salt-services
sudo visudo -c
'
```
`visudo -c` must output `parsed OK`. If not, the sudoers file has a syntax error.

**Step 8 — Configure systemd Restart=always for salt-master**

Run inside WSL2:
```bash
wsl -d Ubuntu-24.04 -e bash -c '
sudo mkdir -p /etc/systemd/system/salt-master.service.d
sudo tee /etc/systemd/system/salt-master.service.d/restart.conf << '"'"'SYSD'"'"'
[Service]
Restart=always
RestartSec=5
SYSD

sudo mkdir -p /etc/systemd/system/salt-api.service.d
sudo tee /etc/systemd/system/salt-api.service.d/restart.conf << '"'"'SYSD'"'"'
[Service]
Restart=always
RestartSec=5
SYSD

sudo systemctl daemon-reload
'
```

**Step 9 — Enable and start salt-master + salt-api**

```bash
wsl -d Ubuntu-24.04 -e bash -c '
sudo systemctl enable salt-master
sudo systemctl start salt-master
sudo systemctl enable salt-api
sudo systemctl start salt-api
'
```

Verify they are running:
```bash
wsl -d Ubuntu-24.04 -e bash -c 'sudo systemctl status salt-master --no-pager && sudo systemctl status salt-api --no-pager'
```

Both must show `active (running)`.

Must show LISTEN on `0.0.0.0:4505`, `0.0.0.0:4506`, and `0.0.0.0:8000`.

This task opens both independent firewall layers on James's machine (.27) for Salt ports. Both commands require an elevated PowerShell prompt.

**Step 1 — Hyper-V Firewall layer (WSL 2.0.9+ added this independent layer)**

Run in elevated PowerShell:
```powershell
Set-NetFirewallHyperVVMSetting -Name '{40E0AC32-46A5-438A-A0B2-2B479E8F2E90}' -DefaultInboundAction Allow
```

NOTE ON SCOPE: The CONTEXT.md recommends targeted ports over blanket allow-all. However, `Set-NetFirewallHyperVVMSetting` only supports `DefaultInboundAction` (all-or-nothing for the WSL2 VM). Per-port Hyper-V rules require `New-NetFirewallHyperVRule` which may not be available on all Windows 11 builds. Use the blanket DefaultInboundAction Allow as the proven approach from RESEARCH.md. The Windows Defender layer below provides the per-port, LAN-scoped security boundary.

Verify:
```powershell
Get-NetFirewallHyperVVMSetting -Name '{40E0AC32-46A5-438A-A0B2-2B479E8F2E90}' | Select-Object DefaultInboundAction
```
Must show `DefaultInboundAction: Allow`.

**Step 2 — Windows Defender Firewall — LAN-scoped inbound rules for Salt ports**

Run in elevated PowerShell:
```powershell
New-NetFirewallRule -DisplayName "Salt Master ZMQ 4505" -Direction Inbound -Protocol TCP -LocalPort 4505 -RemoteAddress 192.168.31.0/24 -Action Allow
New-NetFirewallRule -DisplayName "Salt Master ZMQ 4506" -Direction Inbound -Protocol TCP -LocalPort 4506 -RemoteAddress 192.168.31.0/24 -Action Allow
New-NetFirewallRule -DisplayName "Salt API 8000" -Direction Inbound -Protocol TCP -LocalPort 8000 -RemoteAddress 192.168.31.0/24 -Action Allow
```

These are LAN-scoped (192.168.31.0/24 only) per CONTEXT.md recommendation — consistent with existing firewall patterns in rc-agent's firewall.rs.

Verify:
```powershell
Get-NetFirewallRule -DisplayName "Salt*" | Format-Table DisplayName, Enabled, Action -AutoSize
```
Must show all three rules with `Enabled: True` and `Action: Allow`.

**Step 3 — Verify firewall rules with detailed check**
```powershell
Get-NetFirewallRule -DisplayName "Salt Master ZMQ 4505" | Get-NetFirewallPortFilter | Format-Table LocalPort, Protocol -AutoSize
Get-NetFirewallRule -DisplayName "Salt Master ZMQ 4506" | Get-NetFirewallPortFilter | Format-Table LocalPort, Protocol -AutoSize
Get-NetFirewallRule -DisplayName "Salt API 8000" | Get-NetFirewallPortFilter | Format-Table LocalPort, Protocol -AutoSize
```
Each must show the correct LocalPort and TCP Protocol.
