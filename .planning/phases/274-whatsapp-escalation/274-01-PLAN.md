<task type="auto">
  <name>Task 2: Wire tier5_human_escalation to send EscalationRequest via WS</name>
  <files>crates/rc-agent/src/tier_engine.rs</files>
  <action>
Replace the `tier5_human_escalation` stub (line ~2193) with a real implementation that sends an EscalationRequest to the server via WS.

The function currently takes `event: &DiagnosticEvent` and returns `TierResult`. It needs access to the WS sender. Since the tier engine's `spawn()` function already has access to the WS context (it runs inside rc-agent which has a `ws_tx` for sending AgentMessages), thread the WS sender through:

1. **Change `tier5_human_escalation` signature** to be async and accept the WS sender:
   ```rust
   async fn tier5_human_escalation(
       event: &DiagnosticEvent,
       ws_tx: &tokio::sync::mpsc::Sender<AgentMessage>,
       pod_id: &str,
   ) -> TierResult
   ```

2. **Build the EscalationPayload** from the DiagnosticEvent:
   - `pod_id`: from the `pod_id` parameter (read from config at tier engine spawn)
   - `incident_id`: `uuid::Uuid::new_v4().to_string()`
   - `severity`: derive from trigger type — GameMidSessionCrash/WsDisconnect = "critical", others = "high"
   - `trigger`: `format!("{:?}", event.trigger)`
   - `summary`: build from trigger + pod_state snapshot (e.g. "Game crash on pod-3 during active billing session")
   - `actions_tried`: collect from the tier engine's action log if available, or use a static list like `["Tier 1: deterministic rules", "Tier 2: KB lookup", "Tier 3: single model", "Tier 4: multi-model MMA"]` indicating all tiers were exhausted
   - `impact`: derive from pod_state — if billing active "Customer session impacted", else "Pod offline"
   - `dashboard_url`: `format!("http://192.168.31.23:8080/status")` (server status dashboard)
   - `timestamp`: `event.timestamp.clone()`

3. **Send via WS**: `ws_tx.send(AgentMessage::EscalationRequest(payload)).await`. If send fails (WS disconnected), log at error level but still return the TierResult — the server-side fallback (Plan 02) handles offline scenarios.

4. **Return value**: Change from `TierResult::Stub` to:
   ```rust
   TierResult::FailedToFix {
       tier: 5,
       reason: format!("Escalated to human via WhatsApp (incident {})", incident_id),
   }
   ```

5. **Update all callers** of `tier5_human_escalation` (there are 3 call sites at lines ~1220, ~1318, ~1342) to pass `ws_tx` and `pod_id`, and add `.await` since the function is now async.

6. **Thread `ws_tx` into the tier engine**: The tier engine's `spawn()` function creates a tokio task. Find where the WS sender (`mpsc::Sender<AgentMessage>` or similar) is available in rc-agent's main.rs and pass it to `spawn()`. If the tier engine doesn't currently have access, add it as a parameter to `spawn()` and thread it from main. Grep for where `tier_engine::spawn` is called to find the injection point.

IMPORTANT: Do NOT use `.unwrap()` anywhere. Use `if let Err(e) = ws_tx.send(...).await` with error logging. The WS sender being a `tokio::sync::mpsc::Sender` means `send()` returns a Result.
