  <action>
    FATM-07: Make extend_billing_session atomic — wallet debit + time addition in single DB transaction.

    1. In billing.rs, rewrite extend_billing_session to:
       a. Compute extension_cost_paise from current tier rate (keep existing BILL-04 logic).
       b. Look up driver_id from the billing_sessions table (SELECT driver_id FROM billing_sessions WHERE id = ?).
       c. Begin a sqlx transaction: `let mut tx = state.db.begin().await`.
       d. Call `wallet::debit_in_tx(&mut tx, &driver_id, extension_cost_paise, "extension", Some(&session_id), Some(&format!("Extension {}s", additional_seconds)), None)`.
       e. If debit fails (insufficient balance), return an error — do NOT update allocated_seconds.
       f. In the SAME transaction: `UPDATE billing_sessions SET allocated_seconds = allocated_seconds + ? WHERE id = ?`.
       g. `tx.commit().await` — if commit fails, both debit and time addition are rolled back.
       h. ONLY after successful commit, update the in-memory timer (timer.allocated_seconds += additional_seconds) and reset warnings.

    2. Change the function signature from `async fn extend_billing_session(state, session_id, additional_seconds)` (returning nothing) to return `Result<(), String>` so callers can propagate errors.

    3. In routes.rs extend_billing handler (line 3668), update to:
       a. Call the updated extend_billing_session directly (not via DashboardCommand) since it now needs to return a Result.
       b. Return `{"ok": false, "error": "..."}` on failure (insufficient balance, session not found).
       c. Keep the existing BILL-04 validation (60-3600s, multiple of 60).

    4. The DashboardCommand::ExtendBilling path in handle_dashboard_command should also call the updated function but can log errors since dashboard commands are fire-and-forget.

    Important: Do NOT hold any RwLock across .await (standing rule). Snapshot timer data, drop lock, do DB work, re-acquire lock to update in-memory state.

  <action>
    FATM-10: Enforce a hard floor on the final payable amount after all discounts (coupon + staff discount + group discount) are stacked.

    1. In billing.rs, add a constant:
       ```rust
       /// FATM-10: Minimum payable amount after all discounts stacked. 0 = no floor.
       pub const DISCOUNT_FLOOR_PAISE: i64 = 0;
       ```
       Default 0 (no floor) — venue can change this value. This is a compile-time constant for now; config-based override can come later.

    2. In routes.rs start_billing handler, AFTER all discounts are computed (coupon discount, staff discount, group discount) but BEFORE the final wallet debit:
       a. Compute `total_discount = coupon_discount_paise + staff_discount_paise + group_discount_paise`.
       b. Compute `final_price = base_price_paise - total_discount`.
       c. If `final_price < billing::DISCOUNT_FLOOR_PAISE && billing::DISCOUNT_FLOOR_PAISE > 0`:
          - Cap: `final_price = billing::DISCOUNT_FLOOR_PAISE`
          - Adjust the discount amounts proportionally or simply cap the total discount: `total_discount = base_price_paise - billing::DISCOUNT_FLOOR_PAISE`
          - Log: `tracing::info!("FATM-10: Discount floor enforced — original discount {}p capped to {}p (floor={}p)", ...)`
       d. Use `final_price` for the wallet debit, not `base_price_paise - individual_discounts`.

    3. Also enforce the floor in apply_billing_discount handler (line 19535) for mid-session discount application:
       a. After computing the discount amount, check if applying it would bring the session's effective price below the floor.
       b. If so, cap the discount and log the floor enforcement.

    4. Add `discount_floor_paise` to the billing session's metadata/JSON response so the dashboard knows the floor was applied.

    Important: The floor is the MINIMUM the customer pays — not a minimum discount. If base_price=700 and floor=200, max total discount is 500.
