    1. Add ALTER TABLE migration after the CREATE TABLE for coupons:
       ```sql
       ALTER TABLE coupons ADD COLUMN coupon_status TEXT NOT NULL DEFAULT 'available'
         CHECK(coupon_status IN ('available', 'reserved', 'redeemed', 'released', 'expired'));
       ALTER TABLE coupons ADD COLUMN reserved_at TEXT;
       ALTER TABLE coupons ADD COLUMN reserved_for_session TEXT;
       ```
       Wrap each in a check: `SELECT COUNT(*) FROM pragma_table_info('coupons') WHERE name = 'coupon_status'` — skip if column exists.

    **Coupon reservation (routes.rs):**
    2. Create `async fn reserve_coupon(pool: &SqlitePool, coupon_id: &str, session_id: &str) -> Result<(), String>`:
       - `UPDATE coupons SET coupon_status = 'reserved', reserved_at = datetime('now'), reserved_for_session = ? WHERE id = ? AND coupon_status = 'available'`
       - Check affected rows = 1. If 0, return Err (coupon not available — race condition caught).

    3. Create `async fn redeem_coupon(pool: &SqlitePool, coupon_id: &str) -> Result<(), String>`:
       - `UPDATE coupons SET coupon_status = 'redeemed' WHERE id = ? AND coupon_status = 'reserved'`
       - This is called AFTER billing session commit succeeds.

    4. Create `pub async fn restore_coupon_on_cancel(pool: &SqlitePool, session_id: &str) -> Result<(), String>`:
       - `UPDATE coupons SET coupon_status = 'available', reserved_at = NULL, reserved_for_session = NULL, used_count = MAX(used_count - 1, 0) WHERE reserved_for_session = ? AND coupon_status IN ('reserved', 'redeemed')`
       - Also delete the coupon_redemption row: `DELETE FROM coupon_redemptions WHERE billing_session_id = ?`
       - Make this pub so billing.rs can call it.

    5. In start_billing handler: After validate_and_calc_coupon succeeds, call reserve_coupon BEFORE the billing transaction. After billing transaction commits successfully, call redeem_coupon. If billing transaction fails/rollback, call restore_coupon_on_cancel.

    6. In validate_and_calc_coupon: Add `AND coupon_status = 'available'` to the SELECT query (only available coupons can be validated).

    **Restoration on cancel (billing.rs):**
    7. In the session cancellation/end-early paths (search for BillingSessionStatus::Cancelled and CancelledNoPlayable), after the session status UPDATE:
       - Call `crate::api::routes::restore_coupon_on_cancel(&state.db, &session_id).await` (or extract it to a shared module).
       - If the function is not easily callable across modules, move it to a new `coupon.rs` module or make it a standalone function in billing.rs that does the same SQL.
       - Log: `tracing::info!("FATM-09: Coupon restored for cancelled session {}", session_id)`

    **Background TTL expiry job:**
    8. Add a background task (tokio::spawn in server init, similar to orphan detection pattern):
       - Every 60 seconds: `UPDATE coupons SET coupon_status = 'available', reserved_at = NULL, reserved_for_session = NULL WHERE coupon_status = 'reserved' AND reserved_at < datetime('now', '-10 minutes')`
       - Log count of expired reservations if > 0.
       - Initial delay: 120s (let server stabilize first).

    Important: Do NOT hold locks across .await. The coupon_status transitions use SQL UPDATE with WHERE clause as CAS — no in-memory locking needed.

    1. Add a new route in the appropriate route group (public_routes since webhooks have no staff JWT — use HMAC signature verification instead):
       ```
       .route("/webhooks/payment-gateway", post(payment_gateway_webhook))
       ```

    2. Create the handler `async fn payment_gateway_webhook`:
       a. Parse the webhook payload — expect JSON with at minimum:
          - `transaction_id` (String, required — the gateway's unique payment ID, used as idempotency key)
          - `driver_id` (String, required — who to credit)
          - `amount_paise` (i64, required — amount to credit)
          - `status` (String, required — must be "success" or "captured" to proceed)
          - `signature` (String, optional — HMAC verification placeholder)

       b. **Idempotency check** (reuse pattern from topup_wallet):
          ```sql
          SELECT amount_paise, balance_after_paise FROM wallet_transactions WHERE idempotency_key = ?
          ```
          Bind `transaction_id`. If row exists, return the existing result with `{"ok": true, "duplicate": true, "balance_after_paise": ...}`.

       c. **Status check**: If status is not "success" or "captured", log and return `{"ok": true, "action": "ignored", "reason": "status not success"}`. Do not credit.

       d. **Amount validation**: Reject if amount_paise <= 0 or amount_paise > 10_000_00 (Rs 10,000 cap as safety guard).

       e. **Credit wallet**: Use `wallet::credit_in_tx` inside a transaction:
          ```rust
          let mut tx = state.db.begin().await.map_err(|e| ...)?;
          let (new_balance, txn_id) = wallet::credit_in_tx(
              &mut tx, &driver_id, amount_paise, "gateway_topup",
              Some(&transaction_id), Some("Payment gateway credit"), // use transaction_id as idempotency
          ).await?;
          tx.commit().await.map_err(|e| ...)?;
          ```

       f. Return `{"ok": true, "balance_after_paise": new_balance, "txn_id": txn_id}`.

    3. **HMAC signature verification placeholder**: Add a comment block:
       ```rust
       // TODO: Verify HMAC signature from gateway (Razorpay/Cashfree/etc.)

       // let expected = hmac_sha256(webhook_secret, raw_body);

       ```
       For now, the endpoint is protected by being undiscoverable (no public docs) and the idempotency guard prevents replay damage. Real HMAC verification will be added when the payment gateway is selected.

    4. Log all webhook calls at INFO level with transaction_id, driver_id, amount, and action taken (credited/duplicate/ignored).

    Important: The idempotency_key for wallet_transactions is the gateway's transaction_id — this is the key that makes duplicate webhooks safe. credit_in_tx already accepts idempotency_key via the notes parameter but we need to pass it properly.
