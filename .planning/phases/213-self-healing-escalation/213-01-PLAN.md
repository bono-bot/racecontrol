Functions to implement:

1. **_auto_fix_enabled()** (HEAL-08):
   - Read `auto_fix_enabled` from `$REPO_ROOT/audit/results/auto-detect-config.json` at call time (not startup)
   - `NO_FIX=true` env var overrides config (returns 1)
   - Missing config file = enabled (fail-safe, return 0)
   - jq parse failure = enabled (fail-safe)
   - Export function

2. **_sentinel_gate()** (HEAL-03):
   - Args: pod_ip, tier_name
   - Calls check_pod_sentinels from fixes.sh
   - If blocked: emit_fix "heal" pod_ip "SENTINEL_BLOCK_${tier_name}" "sentinel_active" "blocked"
   - Log: "[HEAL] sentinel block: $pod_ip at $tier_name"
   - Returns 1 on block, 0 on clear
   - Export function

3. **attempt_retry()** (Tier 1):
   - Hypothesis comment: "# Hypothesis: Transient failure -- health check will succeed on retry"
   - 2 retries with 5s sleep between
   - Check: `curl -s -m 5 http://${pod_ip}:8090/health` for valid JSON with status
   - If health returns ok: echo "RESOLVED", return 0
   - If both fail: echo "UNRESOLVED", return 1
   - Export function

4. **attempt_restart()** (Tier 2):
   - Hypothesis comment: "# Hypothesis: rc-agent process crashed or hung -- schtasks restart via rc-sentry will recover"
   - Check rc-sentry alive first: `curl -s -m 5 http://${pod_ip}:8091/health`
   - If rc-sentry dead: echo "UNRESOLVED" (cannot restart without rc-sentry)
   - Send: safe_remote_exec pod_ip 8091 "schtasks /Run /TN StartRCAgent" 10
   - Wait 15s for startup
   - Verify: curl health on :8090
   - If recovered: emit_fix, echo "RESOLVED"
   - If still down: echo "UNRESOLVED"
   - Export function

5. **attempt_wol()** (Tier 3):
   - Read wol_enabled from auto-detect-config.json (same pattern as _auto_fix_enabled)
   - If not enabled: echo "UNRESOLVED" (skip tier, do not block escalation)
   - Call wol_pod() from fixes.sh
   - Wait 60s (WoL boot takes time)
   - Verify: ping then curl health
   - echo "RESOLVED" or "UNRESOLVED"
   - Export function

6. **attempt_cloud_failover()** (Tier 4):
   - Hypothesis comment: "# Hypothesis: Local infrastructure failure -- Bono cloud can serve until local recovery"
   - Notify Bono via relay: POST to http://localhost:8766/relay/exec/run with {"command":"health_check"}
   - Write JSON payload to temp file first (standing rule: bash string escaping safety)
   - If Bono relay reachable: emit_fix with "cloud_failover_requested"
   - Verify: curl Bono VPS health endpoint
   - echo "RESOLVED" if Bono healthy, "UNRESOLVED" if not
   - Check _is_cooldown_active for "fleet" "cloud_failover" before activating (Pitfall 7)
   - Export function

7. **escalate_human()** (Tier 5) (HEAL-04 WhatsApp silence):
   - Args: pod_ip, issue_type, severity (default P1)
   - QUIET severity: return immediately, no WhatsApp (HEAL-04)
   - Check venue_state_detect: if "closed" AND IST hour < 7: defer (no WhatsApp during deep night)
   - Check _is_cooldown_active for pod+issue: skip if in 6h cooldown
   - If sending: _record_alert, build message with pod IP, issue type, IST timestamp, "all recovery tiers exhausted"
   - Call _notify_whatsapp_uday with message
   - Also notify Bono via _notify_bono_ws
   - Export function

8. **verify_fix()** (HEAL-05):
   - Args: pod_ip, issue_type
   - Constructs verify function name: `_verify_${issue_type}`
   - Polls every 10s up to 60s deadline
   - If verify function exists and returns 0: emit_fix "heal_verify" with "verification:PASS", return 0
   - If deadline exceeded: emit_fix "heal_verify" with "verification:FAIL", return 1
   - Define lightweight verify functions for each known issue type:
     - `_verify_crash_loop`: curl health :8090 returns valid JSON
     - `_verify_bat_drift`: safe_remote_exec certutil checksum matches expected
     - `_verify_config_drift`: re-run the specific config key check
     - `_verify_log_anomaly`: curl health :8090 (proxy -- log anomaly resolves if agent is healthy)
     - `_verify_flag_desync`: curl :8080/api/v1/flags and check pod has flags
     - `_verify_schema_gap`: always PASS (schema drift requires migration, not auto-fixable)
   - Export all verify functions

9. **escalate_pod()** (main entry point, HEAL-02):
   - Args: pod_ip, issue_type, severity (default P1)
   - First: check _auto_fix_enabled. If false: log and return 0
   - Billing gate: is_pod_idle. If not idle: emit_fix SKIP_BILLING_ACTIVE, return 0
   - Reset _FLEET_HEALTH_CACHE="" at entry (Pitfall 4)
   - Run 5 tiers in sequence. Each tier: _sentinel_gate first, then attempt function, then verify_fix on success
   - If tier returns RESOLVED AND verify_fix returns 0: log success, return 0
   - If tier returns RESOLVED BUT verify_fix fails: log "fix applied but verification failed", continue to next tier
   - If all 5 tiers exhausted: emit_fix "heal" "all_tiers_exhausted", return 1
   - Export function

10. **attempt_heal()** (HEAL-07 live-sync entry point):
    - Args: pod_ip, issue_type, severity (default from finding)
    - This is the function detectors call after _emit_finding (Plan 02 wires this)
    - Delegates to escalate_pod()
    - Export function

    - Print function list and exit 0
