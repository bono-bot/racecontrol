  <action>
    1. In db/mod.rs run_migrations(), add ALTER TABLE migrations for laps:
       ```sql
       ALTER TABLE laps ADD COLUMN assist_config_hash TEXT
       ALTER TABLE laps ADD COLUMN assist_tier TEXT DEFAULT 'unknown'
       ALTER TABLE laps ADD COLUMN billing_session_id TEXT
       ALTER TABLE laps ADD COLUMN validity TEXT NOT NULL DEFAULT 'valid'
       ```
       assist_tier values: 'pro' (no assists), 'semi-pro' (some assists), 'amateur' (full assists), 'unknown'.
       validity values: 'valid', 'invalid', 'unverifiable', 'suspect'.
       billing_session_id links the lap to the billing session (UX-04 integrity: only laps from billed sessions count).

    2. In lap_tracker.rs, modify the lap recording function:
       a. Accept assist_config (HashMap or JSON string of current assist settings: traction_control, stability_control, abs, autoclutch, ideal_line) as parameter.
       b. Compute assist_config_hash = sha256 of sorted JSON of assist config. This is the immutable evidence fingerprint.
       c. Derive assist_tier from assist config: if all assists off → 'pro', if any assist on → check which → 'semi-pro' or 'amateur'. Simple rule: TC+SC+ABS all off = 'pro'; any one on = 'semi-pro'; ideal_line on = 'amateur'.
       d. INSERT the new columns (assist_config_hash, assist_tier, billing_session_id, validity='valid') with each lap record.

    3. UX-07: In lap_tracker.rs, add `pub async fn mark_laps_unverifiable(db: &SqlitePool, session_id: &str, from_lap: i32)`:
       ```sql
       UPDATE laps SET validity = 'unverifiable' WHERE session_id = ? AND lap_number >= ? AND validity = 'valid'
       ```
       This is called when the telemetry adapter disconnects/crashes mid-session. The caller (event_loop or ws_handler) detects adapter loss and calls this. Log at WARN level: "UX-07: Marked laps >= {from_lap} as unverifiable for session {session_id} — telemetry adapter crashed".

    4. UX-04 integrity gate: In the lap INSERT, require billing_session_id IS NOT NULL. If no billing_session_id is provided, reject the lap with a WARN log. This ensures laps can only be created from billed sessions — no manual entry path exists.

    5. Do NOT create any POST /laps or PUT /laps endpoint. Laps are created ONLY from the telemetry processing pipeline. Verify by grepping routes.rs: no manual lap creation route should exist.
