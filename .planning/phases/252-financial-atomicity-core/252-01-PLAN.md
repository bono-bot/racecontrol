<objective>
Wrap billing start in a single atomic database transaction and add idempotency keys to all money-moving endpoints.

Purpose: Prevent orphaned wallet debits (FATM-01), double charges from duplicate requests (FATM-02), and parallel overspend races (FATM-03).

Output: Modified `start_billing` handler using a single BEGIN IMMEDIATE transaction for wallet debit + session creation. Idempotency key columns on billing_sessions and wallet_transactions. Duplicate-request detection on /billing/start, /topup, /billing/{id}/stop, and /refund endpoints.
</objective>

  **1. Rewrite start_billing handler (FATM-01 + FATM-02 + FATM-03):**

  The current flow is: (1) wallet::debit() in its own tx, (2) billing::start_billing_session() with separate INSERT, (3) UPDATE billing_sessions, (4) on error: compensating refund. Replace with:

  a. Extract `idempotency_key` from request body (`body.get("idempotency_key").and_then(|v| v.as_str())`).

  b. If idempotency_key is present, check for existing session:
  ```rust
  if let Some(key) = &idempotency_key {
      let existing = sqlx::query_as::<_, (String, Option<i64>)>(
          "SELECT id, wallet_debit_paise FROM billing_sessions WHERE idempotency_key = ?"
      ).bind(key).fetch_optional(&state.db).await.ok().flatten();
      if let Some((id, debit)) = existing {
          return Json(json!({ "ok": true, "billing_session_id": id, "wallet_debit_paise": debit, "idempotent_replay": true }));
      }
  }
  ```

  c. After all validation (tier lookup, driver check, discount calculation, balance check), begin a single `BEGIN IMMEDIATE` transaction:
  ```rust
  let mut conn = state.db.acquire().await.map_err(|e| ...)?;
  let mut tx = conn.begin().await.map_err(|e| ...)?;
  // FATM-03: BEGIN IMMEDIATE gives exclusive write lock — prevents parallel overspend
  sqlx::query("SELECT 1").execute(&mut *tx).await.map_err(|e| ...)?;
  ```

  d. Within this single transaction, do ALL of:
  - `wallet::debit_in_tx(&mut tx, ...)` — atomic balance check + debit (per FATM-03)
  - `INSERT INTO billing_sessions (id, ..., idempotency_key, wallet_debit_paise, ...)` — session creation
  - `INSERT INTO billing_events` — created + started events
  - `INSERT INTO wallet_transactions` — already done inside debit_in_tx
  - Mark trial as used (if applicable)

  e. `tx.commit().await` — all-or-nothing

  f. AFTER commit (fire-and-forget): post journal entries via accounting module, create in-memory timer, notify agent, broadcast to dashboards.

  g. On any error before commit: tx is dropped, everything rolls back automatically. No compensating refund needed.

  **Important:** The `billing::start_billing_session()` function does a LOT of in-memory work (timer creation, agent notification, dashboard broadcast) that should NOT be inside the transaction. Split the DB INSERT portion out from the in-memory portion. The DB INSERT goes inside the transaction; the in-memory timer setup + notifications happen after commit.
