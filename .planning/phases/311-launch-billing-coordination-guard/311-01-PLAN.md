<objective>
Make the billing stale-cancel logic game-aware so it does not cancel sessions where the game is actively loading on the pod.

Purpose: Prevents the scenario where billing cancels + refunds a session but the game keeps running, letting the customer play for free. This is revenue loss and a billing integrity bug found during E2E testing.

Output: Modified `tick_all_timers` in billing.rs with game-state-aware stale cancel logic.
</objective>

Replace the Bug #11 stale cancel block (lines 1442-1490) in `tick_all_timers` with game-aware logic:

1. **Modify the SQL query** to also SELECT `pod_id` and `created_at`:
   ```sql
   SELECT id, driver_id, wallet_debit_paise, pod_id, created_at FROM billing_sessions
   WHERE status IN ('pending', 'waiting_for_game')
   AND created_at < datetime('now', '-5 minutes')
   AND ended_at IS NULL
   ```
   Update the tuple type to `(String, String, Option<i64>, String, String)`.

2. **Snapshot the active_games map** before iterating (standing rule: never hold lock across await):
   ```rust
   let game_snapshot: HashMap<String, GameState> = {
       let games = state.game_launcher.active_games.read().await;
       games.iter().map(|(k, v)| (k.clone(), v.game_state)).collect()
   };
   ```

3. **For each stale session, decide action based on status + game state:**

   a. If `status == 'pending'` -> always cancel with refund (no game launched yet).

   b. If `status == 'waiting_for_game'`:
      - Look up `game_snapshot.get(&pod_id)`
      - If game is alive (`Launching` or `Running`) AND `created_at` is less than 10 minutes ago -> **skip** (extend). Log: `tracing::info!("LBILL-02: Extending stale session {} — game {} on pod {} (created {})", session_id, game_state, pod_id, created_at)`
      - If game is alive but `created_at` >= 10 minutes ago -> **cancel with refund** (absolute timeout). Log: `tracing::warn!("LBILL-02: Absolute timeout — cancelling session {} despite game alive on pod {} ({}min)", session_id, pod_id, age_minutes)`
      - If game is dead (no entry, or `Idle`/`Error`/`Stopping`) -> **cancel with refund** (current behavior). Log: `tracing::info!("LBILL-03: Cancelling stale session {} — no active game on pod {}", session_id, pod_id)`

4. **Build a Vec of session IDs to cancel** (excluding extended ones), then do the UPDATE only for those IDs. Use a parameterized IN clause or loop individual UPDATEs (SQLite doesn't support dynamic IN with bind params well — loop is fine for typically 0-2 sessions).

5. **Refund logic** stays the same but only runs for sessions being cancelled (not extended).

6. **CRITICAL: Do NOT hold the active_games read lock across the DB queries.** Snapshot first, release lock, then iterate. This follows the standing rule about locks across `.await`.
