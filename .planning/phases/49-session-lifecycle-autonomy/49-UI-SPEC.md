# Phase 49 — UI Design Contract

> Visual and interaction contract for Phase 49: Session Lifecycle Autonomy.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.

---

## Scope Note

Phase 49 is a pure Rust/backend phase. It ships no new React pages or kiosk
components. The UI-facing surfaces are:

1. **Lock screen HTML** — Rust-embedded HTML/CSS served by lock_screen.rs
   (port 18923, viewed in Edge kiosk mode on each pod)
2. **Win32 overlay** — GDI text painted by overlay.rs (96px bar at top of
   1920px-wide screen)
3. **KioskPodCard fleet status** — existing React component reads end_reason
   from billing session payload; no new component, only new badge text

No new React components, no new pages, no new API routes. This spec governs
the copywriting contract for Rust-rendered UI surfaces and the status badge
text contract for the fleet dashboard.

## Typography

### Lock Screen HTML (Rust-rendered, embedded CSS in lock_screen.rs)

Two weights only: 400 (regular) and 700 (bold). Heading weight moved from
600 to 700 to eliminate the third weight tier (checker fix — Option A applied).

| Role | Size | Weight | Line Height | Notes |
|------|------|--------|-------------|-------|
| Body | 16px | 400 | 1.5 | Status messages, driver name |
| Label | 14px | 400 | 1.4 | Secondary info (tier, time) |
| Heading | 24px | 700 | 1.2 | Primary state label ("Game crashed", "Ready") |
| Display | 48px | 700 | 1.0 | Countdown timer digits only |

Heading at 24px/700 reads bold enough at that size. Display at 48px/700 is
visually justified by the size gap (24px difference) — the two weights are
differentiated by size, not weight variation. All new lock screen states
(crash recovery, auto-end, idle PinEntry) must match these sizes.

Source: Existing lock screen HTML patterns in lock_screen.rs (PinEntry,
SessionSummary).

### Win32 Overlay (GDI text in overlay.rs)

GDI does not use web typography units. Match existing overlay text sizing:
- Status text (new crash/auto-end messages): same GDI font height as existing
  "BILLING PAUSED" text — 18pt, weight Bold, color WHITE (#FFFFFF).
- This is enforced by reusing the existing `draw_status_text()` call in
  overlay.rs rather than creating new text paths.

## Copywriting Contract

### Win32 Overlay — New Messages (overlay.rs)

These appear in the 96px GDI bar at the top of the pod's screen during
active sessions. They overlay the game itself.

| Trigger | Copy | Dismissal |
|---------|------|-----------|
| Game crash detected (SESSION-03, attempt 1) | "Game crashed — relaunching..." | Auto-dismissed on successful relaunch |
| Game crash, attempt 2 in progress | "Relaunching... (2 of 2)" | Auto-dismissed on successful relaunch |
| 2nd relaunch failed, auto-end triggered | "Session ending" | Dismissed when lock screen transitions to PinEntry |
| Billing paused (crash recovery active) | No separate overlay — "Game crashed — relaunching..." covers it | — |

Do NOT show timer countdown during crash recovery — the timer is paused.
Show the same remaining-time field as blank or hidden when billing_paused=true.

### Lock Screen HTML — States (lock_screen.rs)

These are full-screen states rendered by the Rust HTTP server.

#### Idle PinEntry (new show_idle_pin_entry() helper — SESSION-02)

The Heading "Ready" is the primary focal point of the idle PinEntry state.
It is rendered at 24px/700 in #FFFFFF, centered on the card, and is the
first text element the customer reads when approaching the rig.

| Element | Copy |
|---------|------|
| Heading (primary focal point) | "Ready" |
| Subheading | "Scan the QR code on this rig to begin your session" |
| Small text below | "Or ask staff to assign a session" |

This state replaces the old ScreenBlanked state. It must render without
requiring a booking token (token_id = empty string, driver_name = empty).

#### Auto-End Path (SESSION-01 — nobody watching)

Lock screen goes directly to idle PinEntry. No intermediate "Session ended"
screen is shown. The lock screen never shows an auto-end message to the
customer — the auto-end is invisible from the pod's perspective.

### KioskPodCard — end_reason Badge (fleet dashboard)

The end_reason field is added to billing_sessions in SESSION-01. When a
completed session has end_reason set, the fleet dashboard may display it.

| end_reason value | Badge text | Badge color |
|-----------------|------------|-------------|
| 'manual' | (no badge — normal, do not clutter) | — |
| 'orphan_timeout' | "Auto-ended" | #E10600 (Racing Red) |
| 'crash_limit' | "Crash limit" | #E10600 (Racing Red) |
| NULL / absent | (no badge) | — |

Note: The KioskPodCard currently shows ended sessions briefly with
billing.status = 'completed'. The end_reason badge should appear on the
same card during that transition window (typically ~30s). This is additive
to existing status display — no layout changes required.

### Fleet Health Dashboard — new WS message display

SessionAutoEnded WS messages are logged server-side and show in the fleet
health log view. Text format:

| Message type | Display text |
|-------------|-------------|
| SessionAutoEnded (orphan_timeout) | "Pod {N}: session auto-ended (orphan — no game for 5 min)" |
| SessionAutoEnded (crash_limit) | "Pod {N}: session auto-ended (crash limit reached)" |
| BillingPaused | "Pod {N}: billing paused (game crash — relaunching)" |
| BillingResumed | "Pod {N}: billing resumed (game relaunched successfully)" |

These are informational log entries, not modal alerts. They appear in
whatever log/event feed the fleet health dashboard exposes. No new UI
component is required — use the existing event log text style.

### Empty State

Phase 49 introduces no new empty states. The idle PinEntry IS the
"ready / waiting" state — its copy is defined above.

### Error State

| Scenario | Copy shown to customer |
|----------|----------------------|
| Server unreachable during auto-end (force-local-reset path) | Customer sees idle PinEntry ("Ready — Scan the QR code..."). No error text. Staff see log entry. |
| 2nd relaunch failed | Overlay: "Session ending" → transitions to PinEntry. No customer-visible error code. |

Staff-facing errors go to the fleet health log (server-side). Pod-side
lock screen never shows technical error details to the customer.

### Destructive Actions

Phase 49 has no staff-initiated destructive actions (no new buttons, no
confirmations). Auto-end is autonomous — no human confirmation prompt.

## Interaction Contract

### WS Grace Window (SESSION-04) — Lock Screen Behavior

| Duration since WS drop | Lock screen behavior |
|------------------------|---------------------|
| 0–30s | No change — customer sees nothing, session continues |
| >30s | Show existing Disconnected state (no change to existing behavior) |

The 30s suppression is enforced in Rust (main.rs reconnect loop). The lock
screen HTML for the Disconnected state is unchanged — no new copy needed.

### Post-Session Reset (SESSION-02) — Timing

| Path | Sequence |
|------|---------|
| Normal session end | SessionSummary shown → 30s timer → idle PinEntry |
| Orphan auto-end | No SessionSummary → cleanup → idle PinEntry (within 30s) |

The 30s blank_timer is already in main.rs. Only the target changes
(show_blank_screen → show_idle_pin_entry).

## Implementation Notes for Executor

1. All new lock screen HTML must be inline CSS — no external stylesheet
   reference. Lock screen is served by Rust's embedded HTTP server with no
   asset pipeline.

2. Color values in lock screen HTML must use hex literals matching globals.css
   tokens — do NOT reference CSS custom properties (they won't resolve in
   the embedded Rust HTML string context).

3. The Win32 overlay GDI path (overlay.rs) cannot use web fonts. Restrict
   new message text to ASCII. No special characters or emoji.

4. The idle PinEntry state (show_idle_pin_entry) should pass empty strings
   for token_id and driver_name, and 0 for allocated_seconds. The HTML
   template must handle these gracefully — verify no JS errors in Edge kiosk
   mode when these fields are empty.

5. end_reason badge in KioskPodCard is additive only. Do not restructure
   the existing card layout. Render as a small pill (same style as existing
   warning badge) positioned below the session status line.

6. Typography enforcement: lock screen CSS must declare exactly two
   font-weight values — 400 and 700. No 600 (semibold) values anywhere in
   the embedded HTML/CSS string. Heading elements use font-weight: 700.
