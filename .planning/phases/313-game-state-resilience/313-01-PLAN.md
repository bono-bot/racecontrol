  <name>Task 1: GSTATE-01 + GSTATE-03 — Launching hard-cap timeout and stop ACK cleanup</name>

  <action>
Two changes in game_launcher.rs:

**GSTATE-01: Add 180-second hard-cap timeout for Launching state.**

In `check_game_health()` (line ~1295), the existing timeout logic uses `dynamic_timeout_secs` which falls back to 120s (AC) or 90s (others). These are for normal launch timeouts. Add a SECOND check: a hard cap of 180 seconds (3 minutes) that applies REGARDLESS of dynamic_timeout_secs. This catches the case where `launched_at` is None (tracker was created without a timestamp, e.g., from reconnect reconciliation) or where the dynamic timeout was set unreasonably high.

Specifically, after the existing Launching timeout block (line ~1306), add:
```rust
// GSTATE-01: Hard cap — no tracker should stay in Launching for >180s under any circumstance.
// This catches trackers with launched_at=None (created from reconnect) and
// any edge case where dynamic_timeout_secs is unreasonably large.
if tracker.game_state == GameState::Launching {
    if tracker.launched_at.is_none() {
        // No launched_at = likely reconstructed from reconnect. Give it 60s from now,
        // but we can't know when it started. Use a fallback: if no launched_at AND
        // the tracker exists, set launched_at to now on first health check pass.
        // Next check_game_health in 5s will start the real countdown.
        // (We'll handle this below by setting launched_at if None.)
    } else if let Some(launched_at) = tracker.launched_at {
        let elapsed = now.signed_duration_since(launched_at).num_seconds();
        if elapsed > 180 {
            timed_out.push((pod_id.clone(), tracker.sim_type, 180));
        }
    }
}
```

Actually, the simpler approach: in the existing Launching check block (lines 1295-1306), the code already checks `if tracker.launched_at.is_some()`. Trackers with `launched_at == None` are never timed out. Fix this:

1. At the END of the existing `if tracker.game_state == GameState::Launching` block, add an else branch: if `launched_at` is None, collect the pod_id into a separate vec `needs_launched_at` to backfill. After the read lock is dropped, acquire a write lock and set `launched_at = Some(Utc::now())` for those trackers. This way they'll be caught by the normal timeout on the next 5s tick.

2. Add a hard-cap check: regardless of `dynamic_timeout_secs`, if `elapsed > 180`, always add to `timed_out`. The existing check at line 1303 already handles this IF the timeout_secs is <= 180. But if dynamic_timeout_secs is set to e.g. 300, the tracker would survive for 5 minutes. Add after line 1303:
```rust
// GSTATE-01: Hard cap at 180s regardless of dynamic timeout
if elapsed.num_seconds() > 180 && !already_timed_out {
    timed_out.push((pod_id.clone(), tracker.sim_type, 180));
}
```

The cleanest implementation: modify lines 1295-1306 to:
- Keep the existing `dynamic_timeout_secs` check as-is
- After the existing check, add: if NOT already pushed to timed_out AND elapsed > 180, push with 180s
- Handle `launched_at == None`: backfill with current time so next tick catches it

Log with `tracing::warn!("GSTATE-01: ...")` for all hard-cap transitions.

In `stop_game()` (line ~666), after the ACK wait block (lines 704-718), when `result.success` is true (line 706), remove the tracker from `active_games` immediately instead of leaving it in Stopping:

After line 707 (the info log), add:

// GSTATE-03: On successful ACK, remove tracker immediately (don't wait for agent Idle report)
{
    let mut games = state.game_launcher.active_games.write().await;
    if let Some(tracker) = games.get(&*pod_id_owned) {

}
```

Also update pod info to reflect no active game:

```

Keep the existing 30s Stopping timeout spawn (line 758) as a safety net for cases where ACK is not received or fails.

Do NOT use `.unwrap()` anywhere (standing rule). Use `?`, `.ok()`, or match.

<task type="auto">
  <name>Task 2: GSTATE-02 — Smart WS reconnect reconciliation</name>
  <files>crates/racecontrol/src/ws/mod.rs</files>
  <action>
Improve the reconnect reconciliation block at ws/mod.rs lines 417-457. Current behavior:
- Pod reports Running/Launching/Loading: blindly overwrites server tracker state or creates new one
- Pod reports Idle/Stopping/Error: removes any tracker

Problems:
1. If server had Launching (just sent a launch command) and pod reconnects reporting Idle (hasn't processed the command yet), the server removes the Launching tracker — the launch is silently lost.
2. If the pod reconnects with a stale game_state (cached from before disconnect), the server overwrites its more-recent state.

Fix: Make reconciliation smarter by considering the server's current state:

Replace the block at lines 417-457 with:

```rust
// GSTATE-02: Smart reconciliation — merge pod's actual state with server tracker
{
    let mut games = state.game_launcher.active_games.write().await;
    let pod_game_state = pod_info.game_state.unwrap_or(GameState::Idle);
    let server_state = games.get(&canonical_id).map(|t| t.game_state);

    match (server_state, pod_game_state) {
        // Case 1: Server and pod agree (or server has no tracker and pod is idle)
        (None, GameState::Idle) => {
            // Both agree: no game running. No action needed.
        }

        // Case 2: Server has no tracker, pod reports active game
        (None, GameState::Running | GameState::Launching | GameState::Loading) => {
            // Pod knows about a game the server doesn't — create tracker
            if let Some(sim) = pod_info.current_game {
                games.insert(canonical_id.clone(), game_launcher::GameTracker {
                    pod_id: canonical_id.clone(),
                    sim_type: sim,
                    game_state: pod_game_state,
                    pid: None,
                    launched_at: None,
                    error_message: None,
                    launch_args: None,
                    auto_relaunch_count: 0,
                    externally_tracked: true,
                    dynamic_timeout_secs: None,
                    exit_codes: Vec::new(),
                    max_auto_relaunch: 2,
                    playable_at: None,
                    ready_delay_ms: None,
                    billing_session_id: None,
                });
                tracing::info!("GSTATE-02: Created game tracker for pod {} on reconnect ({:?})", pod_info.number, pod_game_state);
            }
        }

        // Case 3: Server has tracker, pod reports active — update state from pod (source of truth)
        (Some(_server_gs), GameState::Running | GameState::Launching | GameState::Loading) => {
            if let Some(tracker) = games.get_mut(&canonical_id) {
                let old_state = tracker.game_state;
                tracker.game_state = pod_game_state;
                if old_state != pod_game_state {
                    tracing::info!("GSTATE-02: Reconciled pod {} game state: {:?} -> {:?} (pod is source of truth)",
                        pod_info.number, old_state, pod_game_state);
                }
            }
        }

        // Case 4: Server has Launching/Running tracker, pod reports Idle
        // The pod is the source of truth for what's actually running.
        // But if server JUST sent a launch (Launching state, <30s ago), the pod may not have
        // processed it yet. In that case, keep the Launching tracker — the normal 180s
        // hard-cap (GSTATE-01) will clean it up if the launch truly failed.
        (Some(GameState::Launching), GameState::Idle) => {
            if let Some(tracker) = games.get(&canonical_id) {
                let keep = match tracker.launched_at {
                    Some(launched_at) => {
                        let elapsed = chrono::Utc::now().signed_duration_since(launched_at).num_seconds();
                        elapsed < 30  // Keep if launched <30s ago
                    }
                    None => true,  // No timestamp = just created, keep it
                };
                if keep {
                    tracing::info!("GSTATE-02: Keeping recent Launching tracker for pod {} despite Idle reconnect (launch may be in-flight)",
                        pod_info.number);
                } else {
                    games.remove(&canonical_id);
                    tracing::info!("GSTATE-02: Removed stale Launching tracker for pod {} (pod reports Idle, launch >30s old)",
                        pod_info.number);
                }
            }
        }

        // Case 5: Server has Running/Loading/Stopping/Error tracker, pod reports Idle
        // Pod is source of truth — game ended while disconnected. Remove tracker.
        (Some(server_gs), GameState::Idle) => {
            games.remove(&canonical_id);
            tracing::info!("GSTATE-02: Removed stale {:?} tracker for pod {} on reconnect (pod reports Idle)",
                server_gs, pod_info.number);
        }

        // Case 6: Pod reports Stopping or Error — update tracker or remove
        (Some(_), GameState::Stopping | GameState::Error) => {
            if let Some(tracker) = games.get_mut(&canonical_id) {
                tracker.game_state = pod_game_state;
                tracing::info!("GSTATE-02: Updated tracker for pod {} to {:?} on reconnect", pod_info.number, pod_game_state);
            }
        }

        // Case 7: No server tracker, pod reports Stopping/Error — create transient tracker
        // so the state is visible on dashboard, it will be cleaned up by normal timeout
        (None, GameState::Stopping | GameState::Error) => {
            if let Some(sim) = pod_info.current_game {
                games.insert(canonical_id.clone(), game_launcher::GameTracker {
                    pod_id: canonical_id.clone(),
                    sim_type: sim,
                    game_state: pod_game_state,
                    pid: None,
                    launched_at: Some(chrono::Utc::now()),
                    error_message: None,
                    launch_args: None,
                    auto_relaunch_count: 0,
                    externally_tracked: true,
                    dynamic_timeout_secs: None,
                    exit_codes: Vec::new(),
                    max_auto_relaunch: 2,
                    playable_at: None,
                    ready_delay_ms: None,
                    billing_session_id: None,
                });
                tracing::info!("GSTATE-02: Created {:?} tracker for pod {} on reconnect (transient)", pod_game_state, pod_info.number);
            }
        }
    }
}
```

Key design decision: the pod is ALWAYS the source of truth for what's actually running on it. The only exception is a very recent Launching state (<30s) where the agent may not have received the command yet — this is protected by the GSTATE-01 180s hard-cap.

Do NOT use `.unwrap()` anywhere (standing rule).
Do NOT hold locks across `.await` (standing rule) — the existing pattern uses a single write lock without await inside, which is correct.
