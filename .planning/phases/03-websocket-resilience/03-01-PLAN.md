  <name>Task 2: Rewrite send_task with WS keepalive ping and app-level round-trip measurement</name>
  <files>crates/racecontrol/src/ws/mod.rs, crates/racecontrol/Cargo.toml</files>
  <action>
Modify the `handle_agent` function in crates/racecontrol/src/ws/mod.rs to add two new capabilities:

**A. WS-level keepalive ping (CONN-01):**

Replace the current `send_task` (lines 54-62) with a `tokio::select!` loop that interleaves:
- `cmd_rx.recv()` -- forwarding CoreToAgentMessage as JSON (existing behavior)
- `ping_interval.tick()` every 15s -- sending `Message::Ping(vec![].into())` WS frame

Use `MissedTickBehavior::Skip` on the ping interval (if channel is busy, skip rather than burst pings).

Handle `None` from `cmd_rx.recv()` by breaking the loop (channel closed = handle_agent exiting).

The `Message::Ping` import is already available from `axum::extract::ws::Message`. For the ping payload, use `vec![].into()` to create an empty `Bytes` -- this avoids needing to add `bytes` as a direct dependency. axum's Message::Ping accepts `Vec<u8>` converted via `.into()`.

If `bytes` is needed as a direct dep, add `bytes = "1"` to `[dependencies]` in crates/racecontrol/Cargo.toml. Check first by trying `vec![].into()` -- axum 0.8 Message::Ping takes `Bytes` which implements `From<Vec<u8>>`.

**B. Application-level round-trip measurement (PERF-03):**

Add a second interval in the same `tokio::select!` loop: `measure_interval` at 30s. When it fires:
1. Generate a ping ID using `std::sync::atomic::AtomicU64` counter (static, like CONN_COUNTER)
2. Record `(id, Instant::now())` in a local `Option<(u64, std::time::Instant)>` variable (only one outstanding at a time -- simpler than HashMap)
3. Send `CoreToAgentMessage::Ping { id }` as JSON via `ws_sender`

In the receive loop (the `while let Some(Ok(msg))` block), add a match arm for `AgentMessage::Pong { id }`:
- Look up the pending Instant for this id
- Compute elapsed
- If >200ms: `tracing::warn!("WS round-trip slow: {} took {}ms (threshold 200ms)", registered_pod_id_or_conn, elapsed.as_millis())`
- If <=200ms: `tracing::debug!("WS round-trip: {}ms", elapsed.as_millis())`
- Clear the pending measurement

To share the pending ping between the send_task and receive loop, use `Arc<tokio::sync::Mutex<Option<(u64, Instant)>>>`. Create it before spawning send_task. send_task writes the (id, Instant) when sending a measurement ping. The receive loop reads and clears it when Pong arrives.

Also update the send_task to send the measurement ping via the ws_sender (it has exclusive ownership of ws_sender). The measurement interval fires inside send_task alongside the keepalive ping.

Add these imports at the top of the file:
```rust
use std::time::Instant;
use tokio::time::{interval, Duration, MissedTickBehavior};
```

DO NOT add a pong timeout (per Claude's Discretion decision in RESEARCH.md) -- the existing is_closed() check in pod_monitor handles dead connections.

Log a trace-level message for each WS keepalive ping sent: `tracing::trace!("WS ping sent (conn_id={})", conn_id)`

CRITICAL: Do NOT manually send Pong frames in response to incoming WS Ping. tungstenite auto-queues pong replies. Manual pong would cause double-pong (RFC 6455 violation). The `_ => {}` arm at the end of the ws_receiver match already handles unknown message types silently.
