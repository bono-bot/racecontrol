**Usage:**
```
bash test/gate-check.sh --pre-deploy          # Before OTA wave 1
bash test/gate-check.sh --post-wave N         # After wave N completes
bash test/gate-check.sh                       # Runs --pre-deploy by default
```

**Structure (follow run-all.sh patterns — color helpers, suite-by-suite reporting):**

```
#!/usr/bin/env bash
# gate-check.sh — Standing rules enforcement gate for OTA pipeline.
# Extends comms-link run-all.sh as a superset. Reads standing-rules-registry.json.
#
# Modes:
#   --pre-deploy    Full pre-deploy gate (before wave 1)
#   --post-wave N   Post-wave verification (after wave N)
#
# Exit codes:
#   0 — all gates passed
#   1 — at least one gate failed (pipeline must rollback)
#   2 — HUMAN-CONFIRM items pending (pipeline must pause)
```

**Pre-deploy mode (`--pre-deploy`) runs these suites in order:**

Suite 0: comms-link E2E framework
- `cd` to comms-link repo (`C:/Users/bono/racingpoint/comms-link`)
- Run `bash test/run-all.sh`
- Capture exit code. If non-zero, report FAIL and set overall failure.
- This is the SYNC-06 requirement: gate-check.sh EXTENDS run-all.sh.

Suite 1: Cargo tests
- `cd` back to racecontrol repo root
- Run `cargo test --workspace 2>&1`
- Capture exit code

Suite 2: Standing rules AUTO checks
- Parse `standing-rules-registry.json` using node (available on James's machine):
  ```bash
  node -e "const r=require('$REPO_ROOT/standing-rules-registry.json'); r.filter(x=>x.type==='AUTO'&&x.check_command).forEach(x=>console.log(x.id+'|||'+x.check_command))"
  ```
- For each AUTO rule: run the check_command, capture exit code
- Report pass/fail per rule ID with color output
- Any failure = overall failure

Suite 3: Diff analysis (pre-deploy specific)
- Check for `.unwrap()` in changed files: `git diff HEAD~1 --name-only -- '*.rs' | xargs grep -n '\.unwrap()' 2>/dev/null | grep -v '#\[test\]' | grep -v '#\[cfg(test)\]'`
- Check for `: any` in changed TS files: `git diff HEAD~1 --name-only -- '*.ts' '*.tsx' | xargs grep -n ': any' 2>/dev/null`
- Check release-manifest.toml exists in deploy-staging: `test -f deploy-staging/release-manifest.toml`

Suite 4: HUMAN-CONFIRM checklist output
- Parse HUMAN-CONFIRM entries from registry
- Print each as a numbered checklist:
  ```
  === HUMAN-CONFIRM: Operator Checklist ===
  [SR-TESTING-001] Visual verification for display-affecting deploys
    [ ] Screens showing correctly on all affected pods?
    [ ] No flicker/misalignment/rendering issues?
  [SR-TESTING-002] Audit what customer sees
    [ ] Window titles checked via tasklist /V?
  ```
- If ANY HUMAN-CONFIRM rules are relevant (display changes in diff, guard enablement in diff), exit with code 2
- If no HUMAN-CONFIRM rules are relevant to the current diff, skip and report "No HUMAN-CONFIRM rules triggered"

**Post-wave mode (`--post-wave N`) runs:**

Suite 0: comms-link E2E (same as pre-deploy)

Suite 1: Build ID verification
- For each pod in the wave, verify build_id matches manifest:
  ```bash
  EXPECTED=$(grep 'git_commit' deploy-staging/release-manifest.toml 2>/dev/null | cut -d'"' -f2)
  # Note: actual pod health check would be done by the OTA pipeline — this just verifies the manifest field exists
  echo "Expected build_id: $EXPECTED"
  test -n "$EXPECTED"
  ```

Suite 2: Fleet health check
- `curl -s http://192.168.31.23:8080/api/v1/fleet/health` (server health endpoint)
- Parse JSON, check all pods in wave show `ws_connected: true`
- Failure = block next wave

Suite 3: Standing rules AUTO checks (same as pre-deploy Suite 2)

**Output format (per suite):**
```
Suite N: {name}... PASS|FAIL
  [SR-XXX-NNN] {summary}... PASS|FAIL
```

```
  FAILED: SR-CODE-003 — .bat files clean ASCII
Suite 3 (diff analysis):      PASS
Suite 4 (HUMAN-CONFIRM):      2 items pending

Overall: FAIL (exit 1)
```

- If both failures AND HUMAN-CONFIRM pending, exit 1 (failures take priority)
