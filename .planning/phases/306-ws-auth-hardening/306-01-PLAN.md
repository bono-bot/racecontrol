# Phase 306 Plan 01: WS Auth Hardening

## Objective

Replace static PSK WebSocket auth with per-pod JWT tokens (24h expiry). Auto-rotate 1h before expiry. Disconnect + WhatsApp alert on invalid JWT. PSK remains as bootstrap fallback.

## Tasks

### Task 1: PodClaims + JWT functions (WSAUTH-01 prep)
Add `PodClaims` struct and `create_pod_jwt()` / `decode_pod_jwt()` to `auth/middleware.rs`. Dual-secret rotation support matching existing `StaffClaims` pattern.

### Task 2: Protocol messages (WSAUTH-01/02)
Add `IssueJwt`, `RefreshJwt` to `CoreToAgentMessage` and `JwtAck` to `AgentMessage` in `rc-common/protocol.rs`.

### Task 3: Server WS auth upgrade (WSAUTH-01/03/04)
Replace `verify_ws_token()` with `authenticate_agent_ws()` that tries JWT first, falls back to PSK. WhatsApp alert on JWT rejection. Issue JWT after PSK bootstrap via `issue_pod_jwt_to_agent()`.

### Task 4: JWT rotation timer (WSAUTH-02)
Spawn per-connection task in `handle_agent()` that sleeps 22h then sends `RefreshJwt` message. No reconnection needed.

### Task 5: Agent JWT handling
Store JWT in `AppState.current_jwt` / `jwt_expires_at`. Handle `IssueJwt` and `RefreshJwt` in `ws_handler.rs`. Reconnect loop uses `?jwt=` when valid, falls back to `?token=` PSK. Clear JWT on 401.

## Done Criteria
- Server issues JWT to pods after PSK auth
- Agent stores and uses JWT for reconnects
- JWT auto-rotates before expiry
- Invalid JWT triggers disconnect + WhatsApp alert
- PSK always works as fallback
