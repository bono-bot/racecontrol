# Phase 307 Plan 01: Audit Log Integrity — SHA-256 Hash Chain

## Objective

Add a SHA-256 append-only hash chain to `pod_activity_log` entries, expand coverage to
include config changes, deploys, billing events, and admin actions, and expose tamper
detection via `GET /api/v1/audit/verify`.

## Architecture Decision

Mutex-serialized hash chain: `std::sync::Mutex<String>` (audit_last_hash) in AppState holds
the last computed hash. Lock is held only for hash computation (no .await inside), released
before async DB INSERT. Callers remain non-blocking.

## Tasks

1. DB migration — ALTER TABLE ADD COLUMN entry_hash TEXT, previous_hash TEXT (idempotent)
2. AppState.audit_last_hash field + DB load on startup
3. activity_log.rs — compute and store hash chain in log_pod_activity()
4. Expand logged actions: billing start/end, admin pod ops, config changes, deploy events
5. GET /api/v1/audit/verify endpoint with chain walk + tamper detection
6. Update SELECT queries to return entry_hash/previous_hash columns
