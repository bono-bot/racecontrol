# Phase 13.1: Pod Fleet Reliability - Research

**Researched:** 2026-03-15
**Domain:** Rust/Axum service hardening, Windows process management, fleet deploy resilience
**Confidence:** HIGH — all findings are based on direct codebase inspection of the exact source files involved

## Summary

Three pods (1, 3, 4) went offline simultaneously on March 15. Root cause analysis revealed six distinct failure modes spread across pod-agent, rc-agent, install.bat, and startup scripts. These failures are well-understood and all have clear, surgical fixes — no architectural rethinking is required.

The most dangerous issue is pod-agent exec thread exhaustion (Issue 1). When `start "" rc-agent.exe` is issued via `/exec`, the cmd.exe process never exits because `start ""` returns to cmd but pod-agent waits for cmd to exit, not for the child. Once all 4 semaphore slots are held by stuck commands, `/exec` stops responding while `/health` continues to report `exec_slots_available: 0`. The fix is a slot-drain recovery mechanism: when all 4 slots report 0 available for more than N seconds, pod-agent actively kills the timed-out cmd processes.

The second cluster of failures involves rc-agent config resolution. The current `load_config()` function searches `["rc-agent.toml", "/etc/racecontrol/rc-agent.toml"]` — both are relative paths or Linux paths. When rc-agent starts from the wrong working directory (DeskIn running from `C:\Program Files\DeskIn`), it cannot find the config. The fix is to add the exe-directory path as the first candidate: `std::env::current_exe().parent() / "rc-agent.toml"`.

The remaining failures (install.bat `&&` chaining, missing pod-agent HKLM Run key, wrong CWD in start-rcagent.bat) are already fixed in the install.bat v2 and start scripts that were written during the debugging session. Phase 13.1 must verify these fixes are correct and deployed.

**Primary recommendation:** Fix pod-agent exec exhaustion in pod-agent/src/main.rs first (highest impact — makes all remote management unreliable when triggered), then fix rc-agent exe-path config resolution in racecontrol/crates/rc-agent/src/main.rs, then wire fleet-health.py into a Windows Scheduled Task.

### Pattern 1: Exec Slot Recovery in pod-agent

**What:** Pod-agent has a global `static EXEC_SEMAPHORE: Semaphore = Semaphore::const_new(4)`. The current implementation uses `try_acquire()` — if all 4 slots are busy, `/exec` returns HTTP 429 immediately. The BUG is that slots get permanently consumed when a long-running command (like `start "" rc-agent.exe`) causes cmd.exe to hold the slot past its timeout.

**Root cause — why kill_on_drop doesn't solve it:**

```rust
// Current code (pod-agent/src/main.rs lines 395-452):
let _permit = match EXEC_SEMAPHORE.try_acquire() {  // acquires slot
    Ok(permit) => permit,
    ...
};
let result = timeout(Duration::from_millis(timeout_ms), async {
    Command::new("cmd")
        .args(["/C", &req.cmd])
        .kill_on_drop(true)   // kills cmd.exe when future is dropped
        .output()
        .await                // awaits cmd.exe exit
}).await;
// _permit dropped here — releases slot
```

When the timeout fires, `kill_on_drop(true)` kills cmd.exe. The permit IS dropped. The slot IS released. **The bug is something else:**

The handler returns `Err(StatusCode::INTERNAL_SERVER_ERROR, ...)` after timeout. The HTTP response is sent. But the actual problem observed ("accepts TCP but never responds") is NOT the semaphore — it's that `cmd /C start "" rc-agent.exe` does NOT block because `start` is a cmd built-in that returns immediately. The cmd.exe exits. So the slot IS released.

**Re-examination of the actual bug:** The 4-slot exhaustion requires 4 CONCURRENT slow commands. The more likely scenario is: pod-agent itself was hung or not running (Issue 4 — no HKLM Run key for pod-agent after reboot), causing connections to be accepted at the TCP level by the OS but never processed.

**HIGH confidence conclusion:** The "accepts TCP but never responds" symptom is more consistent with pod-agent being in a crashed/zombie state where the OS still has the port bound (from a previous instance), not with semaphore exhaustion. The semaphore exhaustion (exec_slots_available: 0) may be a separate, less-common scenario.

**Fix for exec exhaustion (defensive hardening still valuable):** Add a `/exec/drain` or automatic recovery endpoint. When health check shows 0 available slots AND the condition has persisted for > 60s, auto-recover by closing and recreating the semaphore.

**Alternative approach (simpler, HIGH confidence):** The simplest defense against stuck slots is to reduce the default exec timeout from 30s to 10s for most operations, and only allow long timeouts (120s) for explicit download commands. This means a stuck cmd will release its slot in 10s instead of 30s.

**When to use:** Any exec that isn't a download should use 10s default timeout. Deploy downloads still need 120s.

### Pattern 2: rc-agent Config Path Resolution (exe-relative)

**What:** `load_config()` in rc-agent/src/main.rs (line 1824) searches:
```rust
let paths = ["rc-agent.toml", "/etc/racecontrol/rc-agent.toml"];
```
Both are wrong on Windows when CWD is not `C:\RacingPoint`. The exe is at `C:\RacingPoint\rc-agent.exe`, so the config is always next to the exe.

**Fix:**
```rust
fn load_config() -> Result<AgentConfig> {
    // Build candidate paths: exe-directory first (most reliable on Windows)
    let exe_dir_path = std::env::current_exe()
        .ok()
        .and_then(|p| p.parent().map(|d| d.join("rc-agent.toml")));

    let mut paths: Vec<std::path::PathBuf> = Vec::new();
    if let Some(p) = exe_dir_path {
        paths.push(p);
    }
    // CWD fallback (for dev: cargo run from repo root)
    paths.push(std::path::PathBuf::from("rc-agent.toml"));
    // Linux path for testing/non-Windows
    paths.push(std::path::PathBuf::from("/etc/racecontrol/rc-agent.toml"));

    for path in &paths {
        if let Ok(content) = std::fs::read_to_string(path) {
            let config: AgentConfig = toml::from_str(&content)?;
            tracing::info!("Loaded config from {}", path.display());
            validate_config(&config)?;
            return Ok(config);
        }
    }
    Err(anyhow::anyhow!(
        "No config file found. Searched: {}",
        paths.iter().map(|p| p.display().to_string()).collect::<Vec<_>>().join(", ")
    ))
}
```

**Source:** Direct inspection of rc-agent/src/main.rs lines 1824-1838 (HIGH confidence).

### Anti-Patterns to Avoid

- **Semaphore const_new limitation:** `Semaphore::const_new(N)` creates a static semaphore. Once all permits are consumed, you cannot "reset" it without recreating the struct. The correct approach is to track exec slot leaks via `/health` endpoint reporting, not try to mutate the static. A better long-term fix is to use `Arc<Semaphore>` in AppSharedState instead of a static, allowing it to be replaced.
- **Using `&&` in install.bat:** Never — always use separate lines or `&` for unconditional execution when mixing with commands that may fail legitimately (like taskkill on a non-running process).
- **Relative config paths as only fallback:** Never have a production binary that can fail to find config just because CWD is wrong.

## Common Pitfalls

### Pitfall 1: Static Semaphore Cannot Be Recovered Without Process Restart

**What goes wrong:** `static EXEC_SEMAPHORE: Semaphore = Semaphore::const_new(4)` — once a slot is permanently consumed (permit held but never dropped due to a panic or early return bug), there is no way to release it without restarting pod-agent.
**Why it happens:** Rust's ownership guarantees mean the permit must be explicitly dropped. A panic in the handler DOES drop the permit (Rust unwind semantics), but early returns that fail to propagate the permit correctly would not.
**How to avoid:** Verify the permit drop path in exec_command — it IS correct (line 396: `let _permit = ...` — underscore-prefixed variable drops at end of scope regardless of which branch is taken). The slots should self-recover on every command completion.
**Warning signs:** `exec_slots_available: 0` in `/health` response while no commands are actively executing.

### Pitfall 2: `start ""` in cmd.exe Behaves Differently Than Expected

**What goes wrong:** `start "" rc-agent.exe` in cmd.exe is a shell builtin that launches a new window and returns immediately. `cmd /C "start "" rc-agent.exe"` therefore exits immediately with exit code 0. But pod-agent's exec_command runs `Command::new("cmd").args(["/C", &req.cmd])` — so `start ""` commands DO exit the cmd.exe process quickly.
**Why it matters:** The pod_monitor.rs deploy code uses `start /b rc-agent.exe` for the same reason — it exits immediately. This is correct behavior.
**Actual problem:** When pod-agent is NOT running (because of missing Run key), the server TCP port may linger in TIME_WAIT or be unbound, causing connection refused or silent drops.

### Pitfall 3: current_exe() Can Return UNC Paths on Windows

**What goes wrong:** On Windows, `std::env::current_exe()` may return a UNC path like `\\?\C:\RacingPoint\rc-agent.exe`. The parent of this path works correctly in most cases, but `std::fs::read_to_string` on a path derived from a UNC prefix may behave unexpectedly in older Rust versions.
**How to avoid:** Test the config load with `cargo test` using a fixture file in the exe's directory. In practice with Rust 1.93 on Windows 11, `current_exe().parent().join("file")` works correctly.

### Pitfall 4: HKLM Run Keys and Session 1 GUI

**What goes wrong:** HKLM Run keys only fire at USER LOGIN (Session 1). If a pod is logged in and rc-agent crashes mid-session, pod-agent's internal watchdog restarts rc-agent via `watchdog_ensure_running` — but this runs in the pod-agent process's session, which is Session 1 (pod-agent itself was started via HKLM Run key at login). So pod-agent's watchdog restart of rc-agent IS in Session 1. This is correct.
**Warning sign:** The Session 0 problem only occurs if pod-agent is started by a SYSTEM-level task (like the old watchdog scheduled task that was deleted). HKLM Run keys are per-user-login, so they're always Session 1.

### Pitfall 5: Pod 8 Has pod-agent v0.4.0

Per MEMORY.md: "All 8 pods have pod-agent v0.5.0 (Pod 8 has v0.4.0)". Any pod-agent changes in this phase will need to be deployed to all 8 pods including Pod 8. The Cargo.toml shows v0.5.0 — deploying the new build to Pod 8 will upgrade it to v0.5.x simultaneously.

### Pitfall 6: rc-agent Single-Instance Mutex

rc-agent/src/main.rs (lines 186-208) uses a global mutex `Global\RacingPoint_RCAgent_SingleInstance`. When start-rcagent.bat calls `taskkill` first and waits 2 seconds, then launches rc-agent, the mutex will be released by the killed process. But the 2-second sleep may not be enough if the process is in the middle of a cleanup operation.
**How to avoid:** start-rcagent.bat already has a `timeout /t 2` — this is sufficient in practice. The single-instance guard exits quietly (`std::process::exit(0)`) if the mutex is already held, so a race condition just means the new launch exits silently and the old process continues.

```rust
// Source: direct fix for rc-agent/src/main.rs load_config() (lines 1824-1838)
fn load_config() -> Result<AgentConfig> {
    // Primary: exe directory (correct on Windows regardless of CWD)
    let mut search_paths: Vec<std::path::PathBuf> = Vec::new();

    if let Ok(exe_path) = std::env::current_exe() {
        if let Some(exe_dir) = exe_path.parent() {
            search_paths.push(exe_dir.join("rc-agent.toml"));
        }
    }
    // Fallback: CWD (useful for `cargo run` in dev)
    search_paths.push(std::path::PathBuf::from("rc-agent.toml"));
    // Legacy Linux path
    search_paths.push(std::path::PathBuf::from("/etc/racecontrol/rc-agent.toml"));

    for path in &search_paths {
        if let Ok(content) = std::fs::read_to_string(path) {
            let config: AgentConfig = toml::from_str(&content)?;
            tracing::info!("Loaded config from {}", path.display());
            validate_config(&config)?;
            return Ok(config);
        }
    }

    Err(anyhow::anyhow!(
        "No config file found. Searched: {}",
        search_paths.iter()
            .map(|p| p.display().to_string())
            .collect::<Vec<_>>()
            .join(", ")
    ))
}
```
