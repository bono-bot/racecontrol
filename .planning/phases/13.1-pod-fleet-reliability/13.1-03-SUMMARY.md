### Issues During Execution

**1. Pod 8 pod-agent upgrade interrupted**
- **Found during:** Task 1, Step 4 (Deploy hardened pod-agent to Pod 8)
- **Issue:** Standard kill→delete→download→start procedure killed pod-agent with the first step. The process that received the kill command was pod-agent itself. No subsequent commands could be sent. The old binary (v0.5.0) is still on the pod but the process is not running.
- **Impact:** Pod 8 is running without pod-agent until reboot or manual restart. The HKLM Run key IS set, so next login/reboot auto-restarts it.
- **Resolution:** Pod 8 needs a reboot or manual pod-agent start (physical access or command from a privileged machine). After restart, re-run the upgrade using the compound cmd approach or deploy-pod pattern.

**2. Pods 1, 3, 4 exec timeout**
- **Found during:** Task 1, Step 1 (Check PodAgent Run key)

## Next Phase Readiness

**For Pod 8 v0.5.1 upgrade after pod-agent is restarted:**
Use a compound command that creates a restart.bat and schedules it, avoiding the self-kill problem:
```bash
# Write upgrade.bat to pod via /write endpoint, then schedule via schtasks
```

Or simpler: use the start-podagent.bat trick — rename old binary first, then download new, then kill:
```json
{"cmd": "cd C:\\RacingPoint && ren pod-agent.exe pod-agent.exe.bak && curl -s -o pod-agent.exe http://192.168.31.27:9998/pod-agent.exe && taskkill /F /IM pod-agent.exe && start pod-agent.exe", "timeout_ms": 120000}
```

This renames old binary, downloads new, kills old (which self-kills), and starts new — but the start might not execute after kill. Better approach:
```json
{"cmd": "cd C:\\RacingPoint && del /Q pod-agent.exe.bak 2>nul & move pod-agent.exe pod-agent.exe.old & curl -s -o pod-agent.exe http://192.168.31.27:9998/pod-agent.exe", "timeout_ms": 120000}
```
Download first, then kill via start-podagent.bat which does its own kill-and-start.
