1. Create `crates/rc-sentry-ai/src/alerts/ws.rs`:
   - Define `AlertWsState` struct holding `alert_tx: broadcast::Sender<AlertEvent>` (wrapped in Arc)
   - Create `pub fn alerts_router(state: Arc<AlertWsState>) -> axum::Router` with route `GET /ws/alerts` mapped to `ws_handler`
   - `ws_handler` uses `axum::extract::WebSocketUpgrade` to upgrade the connection
   - On upgrade, subscribe to alert_tx (`alert_tx.subscribe()`), then loop:
     - Receive AlertEvent from broadcast rx
     - Serialize to JSON with serde_json::to_string
     - Send as `axum::extract::ws::Message::Text`
     - On send error (client disconnected), break
     - Handle Lagged by logging and continuing
   - Log client connect/disconnect at info level

2. In `crates/rc-sentry-ai/src/main.rs`:
   - Add `mod alerts;` declaration
   - After creating `recognition_tx`, create alert broadcast: `let (alert_tx, _) = tokio::sync::broadcast::channel::<alerts::types::AlertEvent>(256);`
   - If `config.alerts.enabled`, spawn alert engine: `tokio::spawn(alerts::engine::run(recognition_tx.subscribe(), alert_tx.clone()));`
   - Create `AlertWsState` and merge `alerts::ws::alerts_router(alert_ws_state)` into the app router (alongside health, privacy, enrollment, attendance routers)
   - The alert_tx must be created regardless of config.alerts.enabled so the WS endpoint still works (it just won't receive events if engine disabled)
