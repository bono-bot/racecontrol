# Phase 211.1 Plan 01: Venue Shutdown API + Script Summary

**One-liner:** Staff-initiated audit-gated shutdown via POST /api/v1/venue/shutdown — blocks on active billing and P1 audit issues, then SSH-triggers ordered pods->POS->server sequence via venue-shutdown.sh.

## What Was Built

### venue_shutdown.rs (Rust handler)

Three-step safety gate for staff-initiated venue shutdown:

**Step 1 — Billing drain check:** Reads `state.billing.active_timers` and `state.billing.waiting_for_game`. If either has entries, returns `{"status":"blocked","reason":"billing_active","active_sessions":N}`. Does not proceed.

**Step 2 — Pre-shutdown audit:** `tokio::process::Command::new("ssh")` to `bono@192.168.31.27` running `auto-detect.sh --mode quick --no-fix --no-notify` with 120s timeout. Exit code 0 = proceed. Exit code 1 = P1 issues found, block with output tail. Non-zero/timeout = audit error, block.

**Step 3 — Trigger ordered shutdown:** SSH to James runs `venue-shutdown.sh`. Returns `{"status":"shutting_down"}` immediately (script is background on James after SSH returns).

**Error handling:** SSH unreachable = `{"status":"blocked","reason":"james_offline"}`. All errors return structured JSON. No `.unwrap()` anywhere.

**Route:** `POST /api/v1/venue/shutdown` in `staff_routes` (requires staff JWT).

### venue-shutdown.sh (Bash orchestrator)

Ordered shutdown script run by James (.27) via SSH from the API handler:

1. Saves latest auto-detect `summary.json` to `audit/results/pre-shutdown-findings.json` for boot-time fix reference
2. POSTs to `/api/v1/pods/pod_N/shutdown` for pods 1-8 concurrently (`&` background + `wait`)
3. Sleeps 45 seconds for pods to power down
4. SSH `POS@192.168.31.20 "shutdown /s /t 30"` (non-fatal if POS offline)
5. SSH `ADMIN@100.125.108.37 "shutdown /s /t 60"` via Tailscale (server shuts down 60s after command)
6. Notifies Bono via `comms-link/send-message.js`

James (.27) is NOT in the shutdown sequence — stays alive as always.

Log written to `audit/results/venue-shutdown-YYYY-MM-DD_HH-MM.log`.

## Key Decisions

### Why SSH instead of comms-link relay

The plan specified calling `http://localhost:8766/relay/exec/run` but after inspection:
- `/relay/exec/run` sends `exec_request` to **Bono's VPS** over WebSocket — not James's local machine
- `/relay/shell` endpoint runs on James locally but is hardcoded `APPROVE` tier — requires WhatsApp approval from Uday (not suitable for automated audit)
- No registered command exists for `auto-detect.sh`

SSH from server (.23) to James (.27) at `192.168.31.27` is the correct cross-machine execution path: synchronous, returns exit code, no WhatsApp approval needed, consistent with existing patterns in deploy scripts.

### Why both active_timers AND waiting_for_game in drain check

`waiting_for_game` tracks pods where a customer authenticated and paid, but AC hasn't started yet (game pending). A shutdown during this state would lose the customer's session. The billing gate checks both maps — same pattern as the billing gate in game_launcher.rs.

### Pod ID format: pod_N (not pod-N)

The `main.rs` `seed_pods_on_startup` uses `pod_1` through `pod_8` (underscore). The venue-shutdown.sh uses `pod_$i` to match. The `shutdown_pod` handler in routes.rs does map lookup by ID.

## Deviations from Plan

**1. [Rule 3 - Relay approach] SSH used instead of relay custom_command**

- **Found during:** Task 1 implementation — inspecting comms-link relay code
- **Issue:** Plan assumed `custom_command` existed in relay. Grepping confirmed: relay `/relay/exec/run` routes to Bono VPS, not James local. `/relay/shell` is APPROVE-tier. No `custom_command` support.
- **Fix:** Use `tokio::process::Command::new("ssh")` to call James at `192.168.31.27`. Functionally equivalent — synchronous, returns exit code, audit runs on James as intended.
- **Files modified:** venue_shutdown.rs (ssh_runner helper instead of reqwest call)
