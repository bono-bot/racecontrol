# Phase 211.1: Venue Shutdown Button - Context

**Gathered:** 2026-03-26
**Status:** Ready for planning
**Source:** Inserted phase -- user-specified requirements

<domain>
## Phase Boundary

"Shutdown Racing Point" button on kiosk that safely shuts down all venue infrastructure with a pre-shutdown audit gate. James is primary orchestrator; Bono is fallback when James is offline. Blocks shutdown if P1 (critical) issues found. James implements findings on next boot.

</domain>

<decisions>
## Implementation Decisions

### Shutdown Flow (Primary -- James online)
- Staff presses "Shutdown Racing Point" on kiosk UI
- Kiosk calls racecontrol API endpoint (POST /api/v1/venue/shutdown)
- Server triggers James via comms-link relay exec
- James runs auto-detect.sh --mode quick (pre-shutdown audit)
- If P1 issues found: BLOCK shutdown, return findings to kiosk, staff sees error
- If clean: James orchestrates ordered shutdown via fleet exec + SSH
- Shutdown order: Pods (1-8) then POS PC (.20) then Server (.23). James (.27) stays alive.
- Next boot: James reads pre-shutdown findings and implements any fixes

### Shutdown Flow (Fallback -- James offline)
- Kiosk calls racecontrol API then server tries James relay then timeout (5s)
- Server falls back to Bono via cloud API or direct relay
- Bono runs bono-auto-detect.sh (pre-shutdown audit)
- If P1: blocks. If clean: Bono orchestrates shutdown via Tailscale SSH
- Bono writes findings to INBOX.md + pushes for James to pick up on boot
- Next boot: James reads INBOX.md findings and implements fixes

### Kiosk UI
- Button on kiosk staff page (requires staff PIN auth)
- Shows pre-shutdown audit progress (running... / passed / blocked)
- Confirmation dialog: "Shut down all Racing Point systems? James (.27) will stay online."
- If blocked: shows P1 findings with "Fix first" guidance

### Shutdown Sequence
- Step 1: Run pre-shutdown audit (auto-detect quick mode)
- Step 2: Drain active billing sessions (wait or warn)
- Step 3: Shutdown pods 1-8 (parallel, via rc-agent exec: shutdown /s /t 30)
- Step 4: Shutdown POS PC (SSH or exec)
- Step 5: Shutdown server .23 (SSH: shutdown /s /t 60 -- last, after pods confirmed down)
- James stays alive to implement findings on next boot

### Boot-Time Fix Implementation
- James checks for pre-shutdown findings file on startup
- Parses findings JSON and applies safe auto-fixes (APPROVED_FIXES whitelist)
- Logs actions to LOGBOOK.md
- Notifies Bono of boot-time fixes applied

### Claude's Discretion
- API endpoint design (route path, auth requirements, response format)
- Kiosk UI component design (button placement, styling, progress indicators)
- Shutdown timeout values per device type
- Whether to use WoL MAC addresses for "wake all" companion feature (deferred)

</decisions>
