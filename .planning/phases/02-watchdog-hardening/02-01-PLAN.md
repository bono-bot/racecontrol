  <action>
    In crates/rc-agent/src/lock_screen.rs, inside `serve_lock_screen()` at line ~567 (before the POST /pin check), add a handler for GET /health:

    ```rust
    if first_line.contains("GET /health") {
        let current = state.lock().unwrap_or_else(|e| e.into_inner());
        let is_active = !matches!(
            *current,
            LockScreenState::Hidden | LockScreenState::Disconnected | LockScreenState::ConfigError { .. }
        );
        let status_str = if is_active { "ok" } else { "degraded" };
        let body = format!(r#"{{"status":"{}"}}"#, status_str);
        let resp = format!(
            "HTTP/1.1 200 OK\r\nContent-Type: application/json\r\nContent-Length: {}\r\nConnection: close\r\n\r\n{}",
            body.len(), body
        );
        let _ = stream.write_all(resp.as_bytes()).await;
        return;
    }
    ```

    Place this BEFORE the `POST /pin` handler and AFTER the favicon handler. The key insight is that rc-core's check_lock_screen() already checks for HTTP 200 on port 18923 — the /health endpoint returns 200 always (since the server is running), with status "ok" or "degraded" in the JSON body. The HTTP 200 itself confirms lock screen server is alive. The JSON body provides additional state info for future use.

    Per CONTEXT.md decision: "rc-agent exposes a /health endpoint on a local port; rc-core verifies lock screen responsiveness by hitting it via pod-agent /exec curl". The existing PowerShell Invoke-WebRequest check in pod_monitor.rs already works since it checks for HTTP 200 on the root path. The /health endpoint adds a dedicated path for explicit health checking.

    Note: The /health endpoint returns 200 even for "degraded" because the key signal is "lock screen HTTP server is running" not "lock screen is showing the right content". If the server is unreachable (process dead, port not bound), the check returns 0/error, which is the failure signal.

    For testing: add a #[cfg(test)] mod tests block at the bottom of lock_screen.rs. Since the serve_lock_screen function is async and uses TCP, test the health response logic directly by extracting a small helper function `health_response_body(state: &LockScreenState) -> String` that returns the JSON body, and test that function.
