    - `ChildKeyRequest` struct (Serialize): `name: String`, `limit: f64`, `limit_reset: String`.
    - `ChildKeyResponse` struct (Deserialize): `key: String`, `name: Option<String>`, `limit: Option<f64>`.

    - `PodKeyStore` struct (Serialize, Deserialize): `keys: HashMap<String, PodKey>`.
    - `PodKey` struct (Serialize, Deserialize): `api_key: String`, `pod_id: String`, `provisioned_at: String`, `daily_limit_usd: f64`.
    - `PodKeyStore::load(path: &str) -> anyhow::Result<Self>` — load from JSON file, create empty if absent.
    - `PodKeyStore::save(&self, path: &str) -> anyhow::Result<()>` — persist to JSON.
    - `PodKeyStore::get_key_for_pod(&self, pod_id: &str) -> Option<&PodKey>`.
    - `PodKeyStore::set_key(&mut self, pod_id: &str, key: PodKey)`.

    - `KeyProvisioner` struct: `http: reqwest::Client`, `management_key: String`, `store: PodKeyStore`, `store_path: String`.
    - `KeyProvisioner::new(store_path: &str) -> anyhow::Result<Self>`:
      - Reads `OPENROUTER_MANAGEMENT_KEY` from env. This key lives on the SERVER ONLY.
      - Loads PodKeyStore from store_path.
    - `KeyProvisioner::provision_pod_key(&mut self, pod_id: &str, daily_limit: f64) -> anyhow::Result<String>`:
      - Check if key already exists for pod_id. If yes, return existing key (skip provisioning).
      - POST to `https://openrouter.ai/api/v1/keys` with:
        ```json
        { "name": "rc-watchdog-{pod_id}", "limit": daily_limit, "limit_reset": "daily" }
        ```
      - Headers: `Authorization: Bearer {management_key}`, `Content-Type: application/json`.
      - Parse response for `key` field.
      - Store in PodKeyStore with provisioned_at timestamp.
      - Save store to disk.
      - Log at INFO: "Provisioned child key for {pod_id} with ${daily_limit}/day limit"
      - Return the child API key string.
    - `KeyProvisioner::provision_all_pods(&mut self, pod_ids: &[&str], daily_limit: f64) -> anyhow::Result<Vec<(String, String)>>`:
      - Provisions keys for all pods that don't already have one.
      - Returns vec of (pod_id, api_key) pairs.
    - `KeyProvisioner::list_provisioned_pods(&self) -> Vec<&str>`:
      - Returns pod_ids that have keys.

    2. **Security note:** The management key (`OPENROUTER_MANAGEMENT_KEY`) is NEVER stored in code, config files, or pod environments. It exists ONLY as an env var on the server. Child keys (returned by provisioning) are what pods use as their `OPENROUTER_KEY`.

    3. **Deploy-time integration note (for downstream phases):** `deploy-pod.sh` will call `KeyProvisioner::provision_pod_key()` at deploy time. The returned child key is set as `OPENROUTER_KEY` on the pod. This module provides the provisioning logic; the deploy integration is Phase 269+ scope.

    4. Update `crates/rc-mma/src/lib.rs` — add `pub mod keys;`.

  <action>
    1. Create `crates/rc-mma/src/validation.rs` — Model validation gate (MP-09):

    - `ModelComparison` struct (Serialize, Deserialize, Debug, Clone):
      - `top_tier_model: String` (the expensive model ID)
      - `candidate_model: String` (the cheaper model being evaluated)
      - `role: ModelRole`
      - `top_tier_findings: Vec<CompactFinding>` (simplified for comparison)
      - `candidate_findings: Vec<CompactFinding>`
      - `agreement_score: f64`
      - `timestamp: String` (ISO 8601)

    - `CompactFinding` struct (Serialize, Deserialize, Debug, Clone, PartialEq):
      - `severity: FindingSeverity`
      - `root_cause_normalized: String` (lowercase, trimmed, key phrases extracted)
      - `fix_action_normalized: String`

    - `ValidationHistory` struct (Serialize, Deserialize): `comparisons: Vec<ModelComparison>`.
    - `ValidationHistory::load(path: &str) -> anyhow::Result<Self>` — load from JSON, create empty if absent.
    - `ValidationHistory::save(&self, path: &str) -> anyhow::Result<()>`.

    - `ValidationGate` struct: `history: ValidationHistory`, `file_path: String`, `threshold: f64` (from config.model_validation_threshold), `min_comparisons: usize` (default 10).

    - `ValidationGate::new(config: &MmaConfig, file_path: &str) -> anyhow::Result<Self>`:
      - Load history, set threshold from config (default 0.9 per MP-09).

    - `ValidationGate::record_comparison(&mut self, top_tier: &str, candidate: &str, role: ModelRole, top_findings: &[MmaFinding], candidate_findings: &[MmaFinding]) -> anyhow::Result<f64>`:
      - Normalize findings into CompactFinding (lowercase root_cause, trim whitespace).
      - Compute agreement_score:
        - For each top_tier finding, check if a candidate finding has matching severity AND similar root_cause (Jaccard similarity > 0.6 on word tokens).
        - agreement = matched_count / max(top_tier_count, candidate_count).
        - If both empty, agreement = 1.0.
      - Store comparison in history.
      - Persist to disk.
      - Return the agreement score.

    - `ValidationGate::can_switch_model(&self, top_tier: &str, candidate: &str, role: ModelRole) -> bool`:
      - Filter history for this (top_tier, candidate, role) triple.
      - If fewer than `min_comparisons` (10), return false (not enough data).
      - Compute average agreement across all comparisons for this triple.
      - Return true if average >= threshold (0.9 per MP-09).
      - Log at INFO: "Model validation: {candidate} agreement with {top_tier} for {role}: {avg:.2} ({count} comparisons) — switch {'allowed' | 'blocked'}"

    - `ValidationGate::validation_status(&self) -> Vec<ModelValidationStatus>`:
      - Returns summary for each (top_tier, candidate, role) triple:
        `ModelValidationStatus { top_tier, candidate, role, comparison_count, average_agreement, switch_allowed }`.

    2. **Training mode completion (MP-07):**
    - In `engine.rs` (already created in Plan 02), the `run_session()` method already sets `training_mode` in MmaSessionResult from `config.is_training_active()`.
    - Add to the OpenRouter request headers: `X-Training: true` when `config.is_training_active()` returns true. This tags the session in OpenRouter's usage dashboard.
    - Add `training: bool` field to the prompt context so models know they're in training mode and should be thorough rather than cost-optimized.

    3. **Dual-run mode for validation during training:**
    - `ValidationGate::should_run_candidate(&self, config: &MmaConfig) -> bool`:
      - Returns true only if `config.is_training_active()` (during training, run both top-tier and candidate for comparison data).
      - After training, only top-tier runs until validation gate approves the switch.
    - When should_run_candidate() is true, the engine runs each model role twice: once with top-tier model, once with candidate model. The top-tier result is used for the actual diagnosis; both results are fed to record_comparison().
    - Candidate model IDs are read from `MmaConfig` (e.g., `scanner_model` override field). If no override is set, no candidate comparison runs.

    4. Update `crates/rc-mma/src/lib.rs` — add `pub mod validation;` and re-export `ValidationGate`.
