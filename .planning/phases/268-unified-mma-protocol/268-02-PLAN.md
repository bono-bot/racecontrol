    - `MmaEngine::run_session(&self, context: &DiagnosisContext) -> Result<MmaSessionResult, DiagnosisError>`:
      The core orchestration flow:

      **Step 0: Budget check (MP-04)**
      - Estimate session cost: sum of all model estimated_cost_per_call.
      - If `budget.can_start_session(estimated_cost)` returns false, return `DiagnosisError::BudgetExhausted(daily_limit)`.

      **Step 1: Build prompt**
      - Format symptoms from DiagnosisContext into a diagnostic prompt.
      - Include Unified Protocol v3.1 context (MP-06): embed a section with "PROTOCOL CHECKS: Verify against Quality Gate (build integrity), E2E (data flow), Standing Rules (operational), MMA (cross-model)".
      - Include pod_id, layer, tier, started_at for context.

      **Step 2: Query non-thinking models (MP-03 — architecture bugs)**
      - Filter roster: `roster.non_thinking_models()` (Scanner=Qwen3, CodeExpert=DeepSeek V3, SreOps=Devstral 2).
      - Query in parallel via `tokio::join!` or `futures::join_all`.
      - For each successful response, `budget.record_spend(cost)`.
      - Collect findings.

      **Step 3: Query thinking models (MP-03 — execution-path bugs)**
      - Filter roster: `roster.thinking_models()` (Reasoner=DeepSeek R1, Security=Gemini 2.5 Pro).
      - Include findings from Step 2 in the prompt: "Previous models found: {findings}. Trace the execution path to find bugs they missed."
      - Query in parallel.
      - For each successful response, `budget.record_spend(cost)`.
      - Collect findings.

      **Step 4: Check for fallback trigger**
      - If `client.consecutive_failures() >= config.fallback_max_failures`:
        - Log at WARN: "OpenRouter unreachable after {N} failures — falling back to deterministic rules"
        - Call `fallback::diagnose_deterministic(&context.symptoms)`
        - Set `fallback_used = true` on result.
        - Skip fact-checker (deterministic rules are pre-validated).
        - Return early.

      **Step 5: Fact-checker pass (MP-02)**
      - Get the fact-checker model from roster: `roster.fact_checker()`.
      - Build fact-check prompt: "FACT CHECK: Cross-reference these findings against standing rules. Remove findings that contradict known rules. Flag findings where the recommended action is dangerous. Validate severity assignments. Return ONLY findings that pass fact-check."
      - Include all findings from Steps 2+3.
      - Include standing rules summary (embedded in system prompt).
      - Parse response: mark surviving findings as `is_fact_checked: true`.
      - `budget.record_spend(cost)`.

      **Step 6: Build consensus**
      - If 3+ models agree on the same root_cause (exact string match or fuzzy similarity > 0.8), set `consensus_action` to the agreed fix_action.
      - If no consensus, set `consensus_action = None`.

      **Step 7: Return MmaSessionResult**
      - All findings (fact-checked), consensus_action, total_cost, models_queried, duration_ms, training_mode from config, fallback_used.

    2. **Implement `OpenRouterDiagnose` trait** for MmaEngine:
    ```rust
    impl OpenRouterDiagnose for MmaEngine {
        fn diagnose(&self, context: &DiagnosisContext) -> Result<DiagnosisResult, DiagnosisError> {
            // Create a tokio runtime for sync callers (rc-watchdog has no runtime)
            let rt = tokio::runtime::Runtime::new()
                .map_err(|e| DiagnosisError::Other(format!("runtime creation failed: {}", e)))?;
            let result = rt.block_on(self.run_session(context))?;
            // Convert MmaSessionResult -> DiagnosisResult (rc-common type)
            Ok(DiagnosisResult {
                action_id: context.action_id.clone(),
                findings: result.findings.into_iter().map(|f| f.into()).collect(),
                consensus_action: result.consensus_action,
                total_cost_usd: result.total_cost_usd,
            })
        }
    }
    ```
    This is the bridge: rc-watchdog (no tokio) calls `engine.diagnose()` synchronously. The engine creates its own runtime internally.

(3) Standing Rules — does the fix comply with operational rules? (4) MMA — do multiple models agree on the diagnosis?"
    - This is embedded in the system prompt context, NOT a separate API call.
