Create a `POST /api/v1/pwa/game-request` endpoint that lets customers request a game launch from the PWA.

**Request body:**
```rust
#[derive(Deserialize)]
struct GameRequestBody {
    pod_id: String,
    sim_type: SimType,
}
```

**Handler function `pwa_game_request`:**

1. Extract customer identity from JWT claims (use the same auth extractor as other PWA endpoints -- look for `CustomerClaims` or equivalent in the existing routes)
2. Generate a unique `request_id` using `uuid::Uuid::new_v4().to_string()` (uuid crate is already in racecontrol deps)
3. Validate the pod exists in `state.pods` (or equivalent pod registry in AppState)
4. Validate the requested `sim_type` is in the pod's `installed_games` list (if available from PodInfo)
5. Broadcast via `state.dashboard_tx.send(DashboardEvent::GameLaunchRequested { pod_id, sim_type, driver_name, request_id })`
6. Return `Json(json!({ "ok": true, "request_id": request_id }))`
7. On validation failure, return appropriate HTTP status (404 for unknown pod, 400 for game not installed)

**Route registration:**
Add the new route to the PWA/customer router section. Follow the existing pattern for customer-authenticated routes. The route should use the same JWT middleware as other PWA endpoints.

**Important:** Do NOT add a `pending_game_requests` HashMap to AppState for this phase. The request is fire-and-forget from the backend perspective -- the broadcast to dashboard is sufficient. Staff confirmation uses the existing `POST /api/v1/games/pod/{pod_id}/launch` endpoint. Persistence of pending requests is deferred.

If the customer auth extractor doesn't exist or PWA routes don't have JWT auth yet, use the staff JWT auth extractor instead (staff can also initiate from PWA) and add a comment noting customer auth is Phase 82+.
