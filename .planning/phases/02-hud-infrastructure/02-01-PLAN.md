# Plan 02-01: GDI Resource Cache and Component Paint System

## Goal

Eliminate the per-frame GDI font/pen/brush churn in `paint_hud()` (currently 15-17 GDI kernel alloc/free cycles every 200ms) by caching all constant GDI handles in a `GdiResources` struct with RAII cleanup. Then refactor the monolithic `paint_hud()` match-on-index pattern into a `HudComponent` trait + dispatcher, so adding a new HUD section requires implementing one trait and registering it -- no modifications to paint internals.

## Must-Haves (Goal-Backward)

- [ ] GDI font/pen/brush handles are created once at window init (WM_CREATE) and destroyed once at window close (WM_DESTROY) -- zero per-frame allocation for static resources
- [ ] A `GdiResources` struct with `Drop` impl owns all 13 cached handles (9 fonts, 1 pen, 3 brushes)
- [ ] Dynamic-color resources (RPM brush, invalid-lap brush, badge brush) use RAII `TempBrush` wrapper for guaranteed cleanup
- [ ] `HudComponent` trait with `paint(&self, hdc, rect, data, res)` is implemented by 6 section components + 1 RPM bar component
- [ ] `HudRenderer` dispatches paint calls via `Vec<Box<dyn HudComponent>>` -- no match-on-index in paint path
- [ ] All existing overlay visuals (speed, gear, RPM bar, timer, laps, sectors, badges) render identically after refactor
- [ ] Runtime GDI handle counter logs baseline at init and warns on drift
- [ ] No `.unwrap()` in any new production code

**Requirement:** INFRA-01

<what>

**GdiResources** -- created once at WM_CREATE, destroyed at WM_DESTROY via Drop:

```rust
#[cfg(windows)]
struct GdiResources {
    font_label: HFONT,         // 11px bold "Segoe UI"
    font_value: HFONT,         // 22px bold
    font_gear: HFONT,          // 32px bold
    font_speed: HFONT,         // 16px bold
    font_lap: HFONT,           // 18px bold
    font_sector: HFONT,        // 12px bold
    font_sector_label: HFONT,  // 10px normal
    font_unit: HFONT,          // 9px normal
    font_badge: HFONT,         // 9px bold
    pen_divider: HPEN,         // 1px solid rgb(40,40,40)
    brush_bg: HBRUSH,          // rgb(18,18,18)
    brush_rpm_bg: HBRUSH,      // rgb(30,30,30)
    brush_red: HBRUSH,         // rgb(225,6,0)
}
```

`GdiResources::new()` calls `create_font()` and `CreateSolidBrush`/`CreatePen` for all 13 handles.

`Drop for GdiResources` calls `DeleteObject` on all 13 handles.

**TempBrush** -- RAII wrapper for per-frame dynamic brushes:

```rust
#[cfg(windows)]
struct TempBrush(HBRUSH);
impl TempBrush {
    fn new(color: u32) -> Self { Self(unsafe { CreateSolidBrush(color) }) }
    fn handle(&self) -> HBRUSH { self.0 }
}
impl Drop for TempBrush {
    fn drop(&mut self) { unsafe { DeleteObject(self.0 as *mut _); } }
}
```

**WindowState** -- replaces the raw `Arc<Mutex<OverlayData>>` pointer stored in GWLP_USERDATA:

```rust
struct WindowState {
    data: Arc<Mutex<OverlayData>>,
    res: GdiResources,
    // Task 6 will add: renderer: HudRenderer
    // Task 7 will add: gdi_baseline: u32, timer_tick: u32
}
```

Note: `WindowState` is intentionally minimal here. Task 6 adds `renderer: HudRenderer` and Task 7 adds `gdi_baseline: u32` and `timer_tick: u32`. Each task extends the struct and updates all construction sites.

Update WM_CREATE, WM_PAINT, WM_DESTROY handlers to use WindowState.
Update `paint_hud` signature to accept `&GdiResources`.
Remove all per-paint `create_font()` and `DeleteObject(font_*)` calls.
Replace constant-color brushes with cached versions from `res`.
Use `TempBrush` for dynamic-color brushes (RPM, invalid-lap, badge).
</what>

<why>
This is the core of INFRA-01. Caching 13 GDI handles eliminates ~80 kernel alloc/free calls per second. The RAII `Drop` impl makes it impossible to leak handles on window close. The `TempBrush` wrapper makes it impossible to leak dynamic brushes even on early returns.
</why>

### Task 5: Extract compute_layout() pure function with characterization test
**Requirement:** INFRA-02

<what>
Extract the layout computation logic currently inline in `paint_hud()` (lines 504-513) into a pure, testable function:

```rust
#[derive(Debug, Clone, Copy, PartialEq)]
struct SectionRect {
    x: i32,
    y: i32,
    w: i32,
    h: i32,
}

// BAR_HEIGHT is a module-level constant (96px), not passed as parameter.
// The research example `compute_layout(1920, 96)` was approximate --
// the second arg is unnecessary since BAR_HEIGHT is constant.
fn compute_layout(window_width: i32) -> Vec<SectionRect> {
    let section_widths: [i32; 6] = [120, 200, 100, 200, 200, 60];
    let total_content: i32 = section_widths.iter().sum(); // 880
    let start_x = (window_width - total_content).max(0) / 2;
    let mut rects = Vec::with_capacity(6);
    let mut sx = start_x;
    for &w in &section_widths {
        rects.push(SectionRect { x: sx, y: 12, w, h: BAR_HEIGHT });
        sx += w;
    }
    rects
}
```

Add characterization test verifying exact coordinates for 1920px and 800px widths.

Update `paint_hud()` to call `compute_layout(w)` instead of inline math.
</what>

<why>
`compute_layout()` is the foundation of the component system. Each `HudComponent::paint()` will receive a `SectionRect` telling it where to draw. Extracting it as a pure function makes it testable without Win32.
</why>
