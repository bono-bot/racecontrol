**DEPLOY-02: Graceful agent shutdown.**
1. In main.rs, after `SHUTDOWN_REQUESTED.store(true, ...)`, add a shutdown sequence:
   - Read current `active_billing_session_id` from the failure_monitor state (already available via watch channel).
   - If Some(session_id): send HTTP POST to the server at `http://{server_ip}:8080/api/v1/billing/{session_id}/agent-shutdown` with body `{ "pod_id": self_pod_id, "shutdown_reason": "graceful_sigterm" }`.
   - The server endpoint (new) handles: UPDATE billing_sessions SET shutdown_at = datetime('now') WHERE id = ? AND status IN ('active', 'paused_*'). Then call `authoritative_end_session()` which triggers partial refund calculation via `compute_refund()`.
   - If the HTTP call fails (server down), write a local sentinel file `C:\RacingPoint\INTERRUPTED_SESSION_{session_id}.json` with `{ session_id, pod_id, shutdown_at, elapsed_seconds }` for post-restart recovery.
   - Allow up to 5s for the HTTP call (timeout), then proceed with shutdown regardless.

2. Create `POST /billing/{id}/agent-shutdown` in staff_routes (no auth — agent doesn't have JWT; add to public_routes gated by pod source IP check or service key). Handler:
   - Validate session exists and is active/paused.
   - Call `authoritative_end_session()` with end_reason = "agent_shutdown".
   - The existing refund logic in authoritative_end_session handles partial refund.
   - Return 200 with `{ refund_paise, ended_at }`.

**DEPLOY-04: Post-restart session resume.**
3. In main.rs startup (after WS connection established), add a recovery check:
   - Check for local sentinel files: `C:\RacingPoint\INTERRUPTED_SESSION_*.json`.
   - For each: POST to server `/billing/{session_id}/agent-shutdown` (same endpoint as above — idempotent via CAS from FATM-04).
   - Delete sentinel file after successful server notification.
   - Also: GET `/billing/pod/{pod_id}/interrupted` (new endpoint) — server checks billing_sessions WHERE pod_id = ? AND status IN ('active', 'paused_*') AND shutdown_at IS NOT NULL.
   - If server says session is still active (shutdown_at was set but end not processed), the server ends it now.
   - If server says session was already ended, log info and continue.

4. Create `GET /billing/pod/{pod_id}/interrupted` in public_routes (pod source allowed). Returns any active sessions for this pod that appear interrupted (no heartbeat in 5+ min or shutdown_at set). Server can auto-end these during the GET (fire-and-forget cleanup).

5. Add `shutdown_at` column to billing_sessions if not already present: `ALTER TABLE billing_sessions ADD COLUMN shutdown_at TEXT`.

**DEPLOY-05: WS command_id idempotency.**
1. In protocol.rs, add a wrapper approach: do NOT add command_id to every variant (too many variants, breaking change). Instead, create a wrapper struct:
   ```rust
   /// DEPLOY-05: Message-level idempotency wrapper.
   /// If command_id is present, agent deduplicates. If absent (legacy server), no dedup.
   #[derive(Debug, Clone, Serialize, Deserialize)]
   pub struct CoreMessage {
       #[serde(default, skip_serializing_if = "Option::is_none")]
       pub command_id: Option<String>,
       #[serde(flatten)]
       pub inner: CoreToAgentMessage,
   }
   ```
   This is backward-compatible: old messages without command_id deserialize with `command_id: None`.

2. In ws_handler.rs, change the parse target from `CoreToAgentMessage` to `CoreMessage`:
   - `let core_msg: CoreMessage = serde_json::from_str(text)?;`
   - If `core_msg.command_id` is Some, check a `seen_command_ids: HashMap<String, Instant>` stored in the connection state.
   - If command_id already in seen set and age < 5 minutes, log debug "Duplicate command_id {id}, silently acking" and return (skip processing).
   - If not seen, insert into seen set with `Instant::now()`.
   - If command_id is None, process normally (backward compat with old server).
   - Extract `core_msg.inner` and pass to the existing match block.

3. Add periodic cleanup of seen_command_ids: every 60 seconds, remove entries older than 5 minutes. Use a simple counter or timer in the WS receive loop.

4. On the server side (racecontrol), when sending CoreToAgentMessage via WS, wrap in CoreMessage with a generated UUID command_id. Search for all places that serialize CoreToAgentMessage and send via WS sink. Add `command_id: Some(uuid::Uuid::new_v4().to_string())`. Key locations:
   - ws/mod.rs: anywhere that sends to agent_senders
   - billing.rs: BillingTick sends
   - deploy.rs: deploy commands
   Use a helper: `fn wrap_msg(msg: CoreToAgentMessage) -> CoreMessage { CoreMessage { command_id: Some(uuid::Uuid::new_v4().to_string()), inner: msg } }`

Note: uuid crate should already be a dependency (used for idempotency keys in FATM-02). Verify in Cargo.toml.
