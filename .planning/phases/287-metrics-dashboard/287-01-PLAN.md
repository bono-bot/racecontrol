**1. Create `src/lib/api/metrics.ts` — stub API client:**

Define TypeScript types (no `any`):
```
interface MetricPoint { ts: number; value: number }
interface MetricsQueryResponse { points: MetricPoint[]; resolution: string }
interface MetricSnapshot { metric_name: string; pod_id: string; value: number; recorded_at: string }
interface MetricsSnapshotResponse { metrics: MetricSnapshot[] }
type TimeRange = '1h' | '6h' | '24h' | '7d' | '30d'
```

Stub functions (each with `// TODO: Replace with real API call when Phase 286 ships`):

- `fetchMetricNames(): Promise<string[]>` — returns hardcoded array: `["cpu_usage", "gpu_temp", "fps", "billing_revenue", "ws_connections", "pod_health_score", "game_session_count"]`

- `fetchMetricsQuery(metric: string, from: number, to: number, pod?: number): Promise<MetricsQueryResponse>` — generates synthetic time-series: loop from `from` to `to` at interval based on range (60s raw, 3600s hourly, 86400s daily). Value = sine wave (period varies per metric) + small random noise using seeded-ish approach (use `Math.sin(ts * 0.001)` not `Math.random()` so SWR revalidation doesn't cause flicker). Set resolution based on range: raw if <24h, hourly if <7d, daily otherwise. If `pod` param provided, shift the sine phase by `pod * 0.5` so different pods look different.

- `fetchMetricsSnapshot(pod?: number): Promise<MetricsSnapshotResponse>` — returns one entry per metric name with a plausible current value. Use deterministic values (e.g., cpu_usage=45, gpu_temp=62, fps=58, etc.). If pod filter, return only that pod. Otherwise return entries for pods 1-8.

- `timeRangeToMs(range: TimeRange): number` — converts range string to milliseconds.

- Export `metricsApi` object (matching fleet.ts pattern) with all three functions plus `timeRangeToMs`.
