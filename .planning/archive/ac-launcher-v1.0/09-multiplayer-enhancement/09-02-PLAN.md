<objective>
Synchronized billing for multiplayer sessions: billing starts for all participants only when every player is on-track, with a 60-second connection timeout for non-connecting pods.

Purpose: Fair billing -- no player pays for time while others are still loading (MULT-03). A customer whose AC takes 60s to load DirectX should not have a head start on their billing timer. Per locked decision, a pod that fails to connect within 60s is evicted so others can proceed.

Output: Group-aware billing coordinator in billing.rs that holds billing until all group members reach STATUS=LIVE, with 60s timeout eviction.
</objective>

       ```rust
           pub group_session_id: String,
       ```

       - If Some(group_id):
         a. Get or create `MultiplayerBillingWait` entry for this group_id
         b. Insert this pod into `live_pods` and store the `WaitingForGameEntry` in `waiting_entries`
         c. If `live_pods.len() >= expected_pods.len()`, start billing for ALL pods:
            iterate `waiting_entries`, call `start_billing_session()` for each


    6. For `AcStatus::Off` in a group context:
       - End billing for the disconnected pod only (existing behavior works -- `end_billing_session` is per-pod)
       - Remove the pod from `live_pods` in the MultiplayerBillingWait (if present)
       - Do NOT stop billing for other group members
       - If this pod was in `waiting_entries` (never started billing), just remove it

  <action>
    Per locked decision: "60-second connection timeout -- if a pod fails to connect within 60s, marked as disconnected. Other players proceed. Failed player's billing doesn't start."

    1. When the first pod for a group session creates a `MultiplayerBillingWait` entry (step 5 from Task 1), spawn a 60-second timeout task:
       ```rust
       let state_clone = state.clone();
       let group_id_clone = group_id.clone();
       tokio::spawn(async move {
           tokio::time::sleep(std::time::Duration::from_secs(60)).await;
           multiplayer_billing_timeout(&state_clone, &group_id_clone).await;
       });
       ```

    2. Implement `multiplayer_billing_timeout()`:
       - Lock `multiplayer_waiting` for write
       - Get the `MultiplayerBillingWait` entry for this group_session_id (if it still exists -- it may have been removed if all pods connected in time)
       - If it exists and `live_pods.len() < expected_pods.len()`:
         a. Log warning: "Multiplayer billing timeout: {N} pods failed to connect for group {id}"
         b. Remove non-connected pods from `expected_pods` (keep only those in `live_pods`)
         c. If `live_pods` is non-empty, start billing for all live pods (iterate `waiting_entries` for pods in `live_pods`, call `start_billing_session()` for each)
         d. For pods that were in `expected_pods` but NOT in `live_pods`, do NOT start billing -- they timed out
       - If all pods were already live (entry was consumed), do nothing (timeout is a no-op)

    3. Add a `timeout_spawned: bool` flag to `MultiplayerBillingWait` to ensure the timeout is only spawned once (on first pod entry, not on subsequent pods).

    4. When billing starts normally (all pods connected before timeout), the `MultiplayerBillingWait` entry should be removed from the map. The timeout task will then find no entry and do nothing.
