# Plan 02-01: GDI Resource Cache and Component Paint System

<objective>
Eliminate the per-frame GDI font/pen/brush churn in `paint_hud()` (currently 15-17 GDI kernel alloc/free cycles every 200ms) by caching all constant GDI handles in a `GdiResources` struct with RAII cleanup. Then refactor the monolithic `paint_hud()` match-on-index pattern into a `HudComponent` trait + dispatcher, so adding a new HUD section requires implementing one trait and registering it -- no modifications to paint internals.

Purpose: INFRA-01 (GDI cache) and INFRA-02 (component system) are prerequisites for all Phase 3 HUD layout work.
Output: Refactored overlay.rs with cached GDI handles, component-based paint dispatch, runtime leak detection, and full characterization test coverage.
</objective>

`GdiResources::new()` calls `create_font()` and `CreateSolidBrush`/`CreatePen` for all 13 handles.

```rust

    data: Arc<Mutex<OverlayData>>,
    res: GdiResources,
}
```

Update WM_CREATE to construct WindowState with GdiResources::new().
Update WM_PAINT/paint_hud to accept &GdiResources instead of creating fonts per frame.
Update WM_DESTROY to drop WindowState (RAII frees all handles).
Remove all per-paint `create_font()` and `DeleteObject(font_*)` calls.
Replace constant-color brush creation with cached versions from `res`.
Use `TempBrush` for dynamic-color brushes only (RPM, invalid-lap, badge).
No `.unwrap()` in any new production code.
